ERP development on tryton: Part 1

A large project has come to us to create an ERP system for a manufacturer company. As a server, the choice fell on tryton. However, their documentation is bad enough. On the Internet, we managed to find official documentation , pieces of code with a description and the irc log of the tryton channel . This information is very small for the development of any system on tryton. To close this gap, it was decided to document the entire development process, especially the pitfalls that had to be encountered. So let's get started.

Part 1: Install tryotnd, query structure, work with the database
Part 2: Method Structure, Working with Users and Groups, Installing Modules
Part 3: Work with modules of the company (company), contractors (party), product (product) and warehouse (stock).

1. Install and pre-configure trytond


1.1. Server startup

You can install tryton from repositories, however, this path is not suitable for us. So go to their repository . trytond - server sources, tryton - gtk client. We will have a web interface (yii) as a client, so download the trytond server directly . At the moment, the latest version is 3.2.0, and we will consider it.

The server script is in the bin folder and is called trytond. When I tried to start trytond, I fell out with an error accessing the config file in /etc/trytond.conf. The magic key --help prompted that there is an argument -c (- config) with which you can specify the config file. You have to python trytond - /home/username/trytond_path/etc/trytond.conf server so python trytond - /home/username/trytond_path/etc/trytond.conf .

1.2. DB Preparation

The documentation claims support for postgresql, mysql, sqlite. Based on the correspondence of the developers trytond in irc, it is better not to use mysql. Accordingly, the choice fell on postgresql. I did not have to work with this database before, so I used a small tutorial .

We need to install postgresql, create a user there, and be sure to give it rights to create a database: 

 sudo aptitude install postgresql sudo su -l postgres psql create user tryton with password 'tryton'; alter user tryton createdb; \q


1.3. Edit trytond.conf

Getting to edit the config. The first thing we see is the jsonrpc = localhost:8000 line jsonrpc = localhost:8000 . All documentation uses xml-rpc, although it is not included by default. I'm satisfied with json-rpc, so leave it as it is.

Next comes the database configuration. Fill in the appropriate data. 

 db_type = postgresql db_host = localhost db_port = 5432 db_user = tryton db_password = tryton


trytond admin_passwd = tryton administrator password trytond admin_passwd = tryton ; I trytond refused to create a database with an empty admin password.
Subsequently, we will have to work with modules, so auto_reload = True automatic reload of modules auto_reload = True .
logfile = /home/username/trytond_path/log/debug.log where to add logs logfile = /home/username/trytond_path/log/debug.log .

1.4. Logging

Tryton has several levels of logging (DEBUG, INFO, WARNING, ERROR, CRITICAL are indicated in the documentation ). In the source code of the logging library there were such lines: 

 CRITICAL = 50 FATAL = CRITICAL ERROR = 40 WARNING = 30 WARN = WARNING INFO = 20 DEBUG = 10 NOTSET = 0


The larger the number, the less detailed logs. The irc flashed that NOTSET makes the most extended logs, although the documentation does not say a word about it, so I stopped at the DEBUG level.

In order to set the logging level you need to edit trytond / server.py (27 line): level=logging.INFO .

2. The structure of the request, API for working with the database, creating a database


Config changed, now we start the server. I can find a poor description of the server API here , but this is clearly not enough, so we continue to understand further. The network is full of python code to work with the server directly. It looks like this: 

 from jsonrpc import ServerProxy connection = ServerProxy("localhost", "8000", "tryton", verbose=0) name = "model.ir.model.search" model_ids = conn.execute(name, [])


After long attempts to run this code, it turned out that jsonrpc is not a standard pypi library, but trytondâ€™s own developer library, which can be picked out from the client (not to be confused with trytond / protocols / jsonrpc.py - these are different libraries). Download tryton, unpack it, and in the tryton folder we find our cherished jsonrpc.py. Now you can communicate with trytond in python, but we will have a frontend in php, so we continue to dig further.

2.1. Query structure

After a long debriefing, we get the first working code (do not forget that the trytond server should be running): 

 import json import requests url = 'http://localhost:8000/' id = 1 methodname = 'common.db.list' params = '[]' request = json.dumps({ 'id': id, 'method': methodname, 'params': params, }) r = requests.post(url, data=request) print r.json()


This code is already clear and it is not difficult to transfer it to the necessary PL (php, Java ...). Now examine this query.

We will deal with the url request. All actions from the database are done on url localhost:8000/ localhost:8000/ . We work with url to work with the database. localhost:8000/db_name localhost:8000/db_name . It is assumed that the reader is already familiar with json-rpc , so we will not describe the structure of the request.

2.2. API for working with the database

And here is the long-awaited API for working with localhost:8000 localhost:8000 , which managed to get out of trytond / protocols / dispatcher.py and pieces of code from the irc channel:

Method name Options Description
common.server.version [] Check server version
common.db.list [] List of existing databases
common.server.login [user_name, user_pass] Zaloginitsya (returned id and user cookie)
common.db.create [user_id, user_cookie, database_name, password, lang, admin_password] Create DB
common.db.drop [user_id, user_cookie, database_name, password] Delete DB
common.db.dump [user_id, user_cookie, database_name, password] Dump database
common.db.restore [user_id, user_cookie, database_name, password, data, update = False] Restore db


Tryton has anonymous and authenticated requests. For example, common.server.version is an anonymous request, it can be done without logging in to the server. For common.db.create, you can omit the id and cookie, the query will still be executed. For authenticated requests, you must specify an id and a cookie.

2.3. Creating a database

The next step is to create a database that is necessary for the operation of trytond (users, caching ...) and the ERP system itself. The script runs for about a minute: 

 import json import requests url = 'http://localhost:8000/' id = 2 methodname = 'common.db.create' params = [None, None, 'tryton', 'tryton', 'ru_RU', 'tryton'] request = json.dumps({ 'id': id, 'method': methodname, 'params': params, }) r = requests.post(url, data=request) print r.json()


tryton itself will create the tables it needs in the database. It should be noted that tryton will not be able to create tables in an existing empty database even if the user is given the appropriate rights. We called the database tryton, so further requests should be made to the address localhost:8000/tryton localhost:8000/tryton .
