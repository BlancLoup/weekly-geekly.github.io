Proper access to default methods of interfaces through reflection in Java 8, 9, 10

Translator's note: the development of the CUBA framework generates a large number of R & D projects. During one such project, it turned out that we need to call the default interface methods from the proxy classes. Stumbled upon a very useful article, it seems to me that the experience outlined in it will be at least interesting, as much as possible ‚Äî useful to a wide circle of developers.

When it comes to accessing the default methods of interfaces in Java through reflection, googling does not help much. For example, a StackOverflow solution only works in certain situations and not on all versions of Java.

In this article, various approaches to calling the default methods of interfaces through reflection will be considered; this may be necessary, for example, when creating proxy classes.

TL; DR If you can not wait, then all the ways to call the default methods described in this article are available at this link , and this problem has already been solved in our jOOR library.

Proxy interfaces with default methods


The useful API java.lang.reflect.Proxy has been around for a long time, with its help we can do cool things, for example: 

import java.lang.reflect.Proxy; public class ProxyDemo { interface Duck { void quack(); } public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { System.out.println("Quack"); return null; } ); duck.quack(); } }

This code simply prints: 

 Quack

In this example, we created a proxy instance that implements the Duck interface API using InvocationHandler , which, in essence, is just a lambda called for each method of the Duck interface.

The interesting thing will start when we want to add the implementation of the method to the interface and delegate the call to this method: 

 interface Duck { default void quack() { System.out.println("Quack"); } }

Most likely, you want to write this code: 

 import java.lang.reflect.Proxy; public class ProxyDemo { interface Duck { default void quack() { System.out.println("Quack"); } } public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { method.invoke(proxy); return null; } ); duck.quack(); } }

But this will only generate a long stack of nested exceptions (and this is not associated with a call to the implementation of the method in the interface, so it is simply forbidden to do):

Using the Method Handles API


So, Google search gives us that you need to use the MethodHandles API . Well, let's try! 

 import java.lang.invoke.MethodHandles; import java.lang.reflect.Proxy; public class ProxyDemo { interface Duck { default void quack() { System.out.println("Quack"); } } public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { MethodHandles .lookup() .in(Duck.class) .unreflectSpecial(method, Duck.class) .bindTo(proxy) .invokeWithArguments(); return null; } ); duck.quack(); } }

Cool, it seems that earned!

Calling an interface method with no private access


The interface from the example above was neatly arranged so that the calling code had private access to it, i.e. The interface has been nested in the calling class. And what if we have a non-nested interface? 

 import java.lang.invoke.MethodHandles; import java.lang.reflect.Proxy; interface Duck { default void quack() { System.out.println("Quack"); } } public class ProxyDemo { public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { MethodHandles .lookup() .in(Duck.class) .unreflectSpecial(method, Duck.class) .bindTo(proxy) .invokeWithArguments(); return null; } ); duck.quack(); } }

Almost the same code no longer works. We get the exception IllegalAccessException:

Garbage out. If you google more, you can find the following solution, which gains access to the internals of MethodHandles.Lookup through reflection. 

 import java.lang.invoke.MethodHandles.Lookup; import java.lang.reflect.Constructor; import java.lang.reflect.Proxy; interface Duck { default void quack() { System.out.println("Quack"); } } public class ProxyDemo { public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { Constructor<Lookup> constructor = Lookup.class .getDeclaredConstructor(Class.class); constructor.setAccessible(true); constructor.newInstance(Duck.class) .in(Duck.class) .unreflectSpecial(method, Duck.class) .bindTo(proxy) .invokeWithArguments(); return null; } ); duck.quack(); } }

And, cheers, we get:

One of the goals of the Jigsaw project was precisely to prevent such hacks. So, which solution is better? It? 

 import java.lang.invoke.MethodHandles; import java.lang.invoke.MethodType; import java.lang.reflect.Proxy; interface Duck { default void quack() { System.out.println("Quack"); } } public class ProxyDemo { public static void main(String[] a) { Duck duck = (Duck) Proxy.newProxyInstance( Thread.currentThread().getContextClassLoader(), new Class[] { Duck.class }, (proxy, method, args) -> { MethodHandles.lookup() .findSpecial( Duck.class, "quack", MethodType.methodType( void.class, new Class[0]), Duck.class) .bindTo(proxy) .invokeWithArguments(); return null; } ); duck.quack(); } }

Conclusion


To understand all this is a bit complicated.

  • In Java 8, the best working approach is a hack that fits inside the JDK through access to the package-private constructor of the Lookup class. This is the only way to uniformly invoke interface methods with both private access and non-private access from any class.
  • In Java 9 and 10, the best way is to use Lookup.findSpecial() (does not work in Java 8) or MethodHandles.privateLookupIn() (the method does not exist in Java 8). The latter approach should be used if the interface is in another module. This module should provide an interface for calling from the outside.
