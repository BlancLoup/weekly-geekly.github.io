Altaro VM Backup: Backup Hyper-V and VMware Virtual Machines

Deploying virtual machines is becoming common practice for companies of all sizes. With the introduction of virtualization and the growing popularity of the cloud model, the corresponding backup tools are also evolving, covering more and more virtual environments, operating systems and applications. So how to organize a backup of the virtual environment at the proper level? How hard is it?


Until now, there is a perception that, with all its advantages, virtualization makes backup much more difficult. Perhaps this is one of the reasons why companies ignore the need to protect the virtual environment: according to information from foreign analytical agencies, backup procedures cover less than 70% of virtual servers.

In fact, everything is not so difficult. Regardless of the hypervisor used, such as Hyper-V or VMware, the approaches to backing up virtual machines (VMs) are the same. It can be performed at the host level (or, as it is also called, at the hypervisor level), or at the VM level. Currently, a variety of products of the most diverse class are offered for backing up virtual environments. Some companies use specialized solutions designed specifically for a specific hypervisor, others prefer universal products.

The recovery process in a virtual environment has its own specifics. Instead of restoring the entire VM image, most backup products allow you to restore a single file or group of VM files. And the backup itself can be stored in the cloud, in a remote data center. This method is becoming increasingly popular. In this case, data is usually copied locally and then replicated to the cloud (Cloud Backup). Redundancy in the cloud does not solve all the problems of local protection and high availability, but protects against accidents (DR).

New features of backup products help to more effectively and quickly restore virtual machines, applications and data in case of failure, to find the necessary balance between the allowable recovery time (Recovery Time Objective, RTO), allowable data loss (Recovery Point Objective, RPO) and the cost of the solution. Below we will discuss one of these solutions.

Installation process


Installing Altaro VM Backup is not easy, but very simple. Just run the installer, click "Next" a couple of times, and that's it. No additional components need to be installed. After the process is completed and Altaro VM Backup is launched, a connection is established with an instance of the program on one of the servers, for example, localhost. The required port (35107) is already registered in the interface.
The next step is to connect with

  • Hyper-V host or failover cluster.
  • ESXi host (if vCenter Server is unavailable)
  • vCenter Server



For example, let's choose an ESXi host. The connection will require a name, hostname, account, and password. Ports 443 and 902 (vSphere VDDK) must be open. After the connection with the host is established, you need to choose a place for data backup - local (USB, eSATA, iSCSI) or a network device. Of course, for reliability, it is preferable to store backups on removable media or on another site, for example, to set up replication over the global network. Stored VM data is encrypted.

Backup Verification


How to make sure that the backup can really recover? This is often simply forgotten. In Altaro VM Backup, two verification methods are proposed - by checksums and by actual recovery of VMs. The first, Backup Verification, allows you to simply verify the integrity of the data on the media. The second, Sandbox Restore, will allow you to sleep peacefully. Both options can be planned to automate this process.



Reports window shows backup status, verification result and restore tasks



Setting up and planning Sandbox Restore

Sandbox Restore restores VM under a temporary name. The administrator can start it and make sure that it loads. This will not affect the operation of other VMs.
