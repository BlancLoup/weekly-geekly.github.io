Promo codes.apk


Nowadays, so-called ‚Äúpromotional codes‚Äù are popular with many stores, according to which buyers can get a discount on goods. On average, the customer base of stores using such ‚Äúinnovations‚Äù consists of buyers aged from 16 to 30 years. People of this age group practically do not part with their smartphones. On the face of the relevance of mobile applications, which is growing every year. This brought us and the guys to the idea of ‚Äã‚Äãcreating an application for the distribution of these same promotional codes.

Concept and marketing


After a brief discussion, we stopped at the development of a maze in which the user must find a promotional code. The number of promotional codes, as well as the frequency of maze generation, will depend on the customer. Each user can find a promo code in the labyrinth, but only the first N clients will receive it.

User benefits:


  • the application is free, the client does not have to pay for anything;
  • fascinating "race" for the promo code.

Customer benefits:


  • an increase in the client base due to the original game distribution format of promotional codes;
  • minimal development and hosting costs;
  • being the owners of the application, the possibility of introducing advertising.

Benefits for developers:


  • with the existing template, rapid development for a specific store, by changing the design and minimal changes in mechanics;
  • a steady income in the form of maintaining the health of the hosting and troubleshooting application operation problems;
  • the possibility of introducing advertising (with the consent of the customer).


Unity3D as the main development tool


When choosing a game engine, we stopped at Unity3D . There were several reasons for this:

  • free version (if your organization has a salary of less than $ 100,000 per year);
  • wide functionality of the free version;
  • we already had little experience in the development of this engine.

Data storage


When there was a question about the storage of data, such as: the name of the shares, numbers of promo codes, a map of the maze, etc., we came to the conclusion that storing data locally is at least silly.
First, the volume of the application increases, which clogs the memory of the device, and, secondly, there will be a problem with the update (you would have to do a new build forever and upload it to the site). So we came to the conclusion that the game will work on the principle of ‚Äú Client-Server-DB ‚Äù.



Why not ‚Äú Client - DB ‚Äù? - Because in this case, each user will have direct access to the database, which reduces its security to zero.

Since the database will be located remotely, we need hosting. For the prototype of our game, we chose a free hosting. The main criteria were the provision of two databases at once and the ability to edit data through PhPMyAdmin . Due to the meager experience with the database, the latter has become the key for us.

Access to the database


Since the game accepts the principle of ‚Äú Client-Server-DB ‚Äù - it is logical that the client should contact the server so that he in turn makes a request to the database. Data is returned in the reverse order. The access from the client to the server is made using the WWW and WWWForm already existing in Unity (see the Unity API Scripting for more details) using C # , and from the server to the database using a regular php script. For the distribution of data that is taken from the database, JSON was used in Unity.

‚ÄúWhat is the date today?‚Äù


In each row of the table, we used id as an identifier, according to which we selected from the entire table what interests us. It ranged from 1 to 31, depending on the date of the month. As you already understood, a unique maze, a promo code, etc. was ready for each id . The check was carried out through a short php script that returns a number in a specific time zone (so that everyone has the same number).

The syntax was implemented in such a way as not to overload the database with unnecessary queries.
At first we addressed a script with a date that returned the day of the month. After that there was a check of the local number (which was saved during the last query to the database) with the number that the script returned. If they are the same, then we don‚Äôt refer to the database, but simply use local data (what we took from the previous reference to the database), and if the dates differ, then we refer to the database, and replace the local number with what the script returned with date.
