BEMIT: The next step in the development of BEM according to Harry Roberts

Everyone who has been following me or my work for some time undoubtedly knows that I am a huge fan of the BEM naming convention. What I will discuss in this article is not an alternative or other agreement on the names of BEM, but an addition to it: small additions that will raise BEM one step higher. This extended BEM syntax has been dubbed BEMIT, as it includes some paradigms and patterns from the (yet unpublished) Inverted Triangle CSS Architecture (ITCSS). BEM + ITCSS = BEMIT.

Recall what BEM is - it works by breaking down all classes in the code base into three groups:

‚Ä¢ Block: The only root element of a component.
‚Ä¢ Element: An integral part of the Block.
‚Ä¢ Modifier: Option or block extensions.

Blocks, Elements and Modifiers: BEM. Absolutely any class in the project falls into one of these categories, and that is why BEM is so good - it is incredibly simple and straightforward.

The meaning of BEM is to make your markup more transparent and understandable. BEM tells developers how classes relate to each other, which is important in complex or multi-level DOMs. For example, if I asked you to remove all user-related classes in the following HTML fragment that you would get rid of? 

<div class="media user premium"> <img src="" alt="" class="img photo avatar" /> <p class="body bio">...</p> </div>


Definitely, we would start with the user, but everything below it will be more likely a guess, albeit a reasonable one. However, if we rewrite this fragment with BEM: 

 <div class="media user user--premium"> <img src="" alt="" class="media__img user__photo avatar" /> <p class="media__body user__bio">...</p> </div>


Here we see for sure that user, user - premium, user__photo and user__bio are related to each other. We also see that media, media__img and media__body are also interconnected, and avatar is a lonely Block, which has no Elements and Modifiers.

This level of detail on our classes is great. It allows us to make safer and more informed decisions about what should be done and how to use, reuse, change or delete items.

The only disadvantage of BEM is that it only tells which classes to use in relative terms, and also how the classes are related to each other. He does not make it clear how such elements behave, work, or should be implemented in a global non-relative sense.

From this point of view, I decided to expand BEM and make BEMIT. BEMIT does not add new types of classes - we still only have Blocks, Elements or Modifiers, but it also adds information about usage and state.

Namespaces



The most common types of namespaces are c- for Components, o- for Objects, u- for Utilities, and is- / has- for states (this is described in more detail in the article I refer to). With this in mind, the above HTML should be rewritten as follows: 

 <div class="o-media c-user c-user--premium"> <img src="" alt="" class="o-media__img c-user__photo c-avatar" /> <p class="o-media__body c-user__bio">...</p> </div>


From this, I see that we have a reusable abstraction in a media object (o-media *) and two implementation components (c-user * and c-avatar). These classes still remain Blocks, Elements, or Modifiers: no new classification has been added to them, but a new level of value has been added.

These namespaces are associated with levels contained in an inverted triangle-shaped CSS architecture, which means that each class has a specific place in our project (and in our file system).

Responsive suffixes


The next thing BEMIT adds to traditional BEM naming is responsive suffixes. These suffixes take the format @ <checkpoint>, and tell us this class in the following size: 

 <div class="o-media@md c-user c-user--premium"> <img src="" alt="" class="o-media__img@md c-user__photo c-avatar" /> <p class="o-media__body@md c-user__bio">...</p> </div>


Here we have o-media @ md, which means that the media object will be at this control point. Other possible examples are:

‚Ä¢ u-hidden @ print - utility class that hides items when printing.
‚Ä¢ u-1/4 @ lg - a utility that displays objects in a quarter-width at a large reference point
‚Ä¢ o-layout @ md - layout object in the middle control point.

The @ sign is a logical way for a person to indicate conditional states. It allows developers to learn about any potential permutations or the appearance of this part of the UI in question at a glance.

Note: Do not use the @ symbol in the CSS file as follows: 

 @media print { .u-hidden\@print { display: none; } }


Health check


With such strict and clear patterns in our HTML, we can do many things. First, and most obviously, we can write significantly more expressive and rich code for our colleagues, which means that they can understand and understand the state and form of the project much faster. They will also be able to contribute in a more consistent way.

But another funny side effect is that we can run a visual health check for our UI. Using subscript selectors, we can denote, and therefore visualize the overall appearance of the page, based on the types of classes it contains: 

 /** * Outline   . */ [class] { outline: 5px solid lightgrey; } /** * Outline  BEM-. */ [class*="__"] { outline: 5px solid grey; } /** * Outline  BEM . */ [class*="--"] { outline: 5px solid darkgrey; } /** * Outline    . */ [class^="o-"], [class*=" o-"] { outline: 5px solid orange; } /** * Outline    . */ [class^="c-"], [class*=" c-"] { outline: 5px solid cyan; } /** * Outline   " ". */ [class*="@"] { outline: 5px solid rosybrown; } /** * Outline   "Hack classes". */ [class^="_"] { outline: 5px solid red; }


Of course, this is not a panacea - some fragments can be Components and Elements at the same time - but if we write classes in a selective order (i.e., in order from less important to more important, and thus, the hardest is at the end), we can get a good visual snapshot of the appearance of any particular page.

We can run a health check in various ways, but the simplest is to enclose the entire volume in the Scope class: 

 .s-healthcheck { ... /** * Outline all Responsive classes. */ [class*="@"] { outline: 5px solid rosybrown; } ... }


... which can then be added to the html element when you need to enable validation: 

 <html class="s-healthcheck">


Finally


We have a few simple additions to BEM that turn it into BEMIT: adding information to the beginning and end of a standard Block, Element, or Modifier class to provide us with more data on how classes behave in a non-relative sense. Some more examples: 

 .c-page-head {} @media screen and (min-width: 15em) { .u-text-center\@sm {} } .o-layout__item {} @media print { .u-color-black\@print {} }
