Report on the Exchange environment using Powershell (now on Exchange 2013)


On January 19, a new version of the script was released that provides visual information on the state of the Exchange infrastructure. Steve Goodman added support for Exchange 2013 in version 1.5.6. If you haven‚Äôt met with this useful tool yet, we recommend doing it today.
The script collects a lot of useful information on the Exchange infrastructure and displays it in an html file. You can also automate the receipt of reports by mail.
The list of what is included in the report is quite large - you can find it under the cut.

The script itself can be downloaded here .
Under the cut is a small translation of the author's post about the script.

Exchange administrators often need a document that gives an overview of the current Exchange infrastructure. Word and Visio are not particularly suitable, as they do not allow keeping information up to date or simply keeping track of changes.

The proposed script allows you to automatically create a report for servers and DAG (database availability groups) in Exchange 2003, 2007, 2010 and 2013, namely:
‚Ä¢ Total number of servers per version and SP Exchange
‚Ä¢ Total number of mailboxes per version and SP Exchange
‚Ä¢ Total number of Exchange roles in your IT infrastructure
‚Ä¢ Site breakdown for the following parameters:
o Mailboxes on the site
o Exchange servers, version, service pack and version, service level, installed roles, OS version and service
‚Ä¢ A breakdown for each Database Availability Group, including:
o Name of each DAG, number of members and their list
o Database information:
- Name
- The number of mailboxes per database and their average size
- The number of archive mailboxes per database and their average size - shown only if the database is included in the archive mailboxes
- Database size and free space
-% free space in the database and logical disk
- Time and date of the last backup (new) - shown if at least one DAG database has a full backup
- Circular Logging status (new) ‚ÄîDisplayed if cyclic logging is enabled for at least one DAG database.
- Server where the active copy is located
- List of servers with active copies and the number of copies
‚Ä¢ Breakdown by non-DAG databases, including Exchange 2007 and 2003 databases with information about the database and the name of the Storage Group (where applicable).

The output is such a report (I bring here a small part of it).


It is worth noting that the script does not allow to get detailed information about Exchange 2007/2003 CCR / SCC clusters, but they are shown as ClusMBX in the final report. Also at the moment, the script does not show information about Public folders, but if someone is interested, the author promises to add a couple of functions; and, of course, you yourself can make any changes to the code.

To execute the script, you need Exchange Management Shell (the latest version for your environment with Powershell 2.0) and the ability to receive information about AD sites, Exchange servers, mailboxes, DAG and DB. It uses WMI to extract OS information and identify Exchange 2007 clusters and calculate the size of the Exchange 2007 database and Remote Registy calls to get information about service packs. Any Exchange administrator will be able to perform these tasks.

Running the script is easy - you just need to clarify where to write the HTML file: 

.\Get-ExchangeEnvironmentReport -HTMLReport c:\report.html


If you want to receive reports by mail, the following parameters are available right in the script for sending the report: 

 .\Get-ExchangeEnvironmentReport -HTMLReport c:\report.html -SendMail:$true -MailFrom:you@example.com -MailTo:you@example.com -MailServer:smtp.example.com


Finally, to set the schedule for sending the report daily, add a parameter ‚ÄìScheduleAs, for example: 

 .\Get-ExchangeEnvironmentReport -HTMLReport c:\report.html -SendMail:$true -MailFrom:you@example.com -MailTo:you@example.com -MailServer:smtp.example.com -ScheduleAs:DOMAIN\user


After creating the report, it will try to create a task and will ask you to enter the password of the user for whom you selected to send reports (via schtasks.exe).
