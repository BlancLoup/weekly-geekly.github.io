Play using math graphs instead of graphs



This screenshot shows you a seemingly ordinary pixel-based game. However, not all so simple.

In some places, the relief of the earth resembles a sinusoid, and the bullets resemble two symmetrical graphs of the root of x.

In fact, everything that you see on the screen in one way or another relates to mathematics, mathematical curves and graphs.

Trees


I confess, the formula for creating the silhouette of trees to withdraw or pick up I could not. But, in order not to violate the basic concept of the whole game and the rules not to use any .png and .jpg format files, I used one trick. I used fractals to create trees.


An example of a fractal tree

Most likely you will agree that the fractal trees themselves look quite boring. If you add a few elements of randomness, for example, 2 branches may not necessarily grow, but also 3, or 1, or not at all. Also, you can do not always the same angle of inclination.

Of course, one could make an ordinary machine pseudorand , which was based on computer ticks, but the following idea seemed to me more interesting:

‚ÄúWhat if we give each tree a certain number (sid) from which pseudo-random numbers will be calculated that affect the parameters of the tree?‚Äù

Fortunately, in c ++ there is a separate library responsible for the pseudorand.

Some more code


Above I gave the code only for generating textures. In this subtitle I will describe the code of the game itself. All the code is on GitHub, the link to the project is in the conclusion.

Player


The player has two different methods update - spaceUpdate and planetUpdate. Accordingly, spaceUpdate updates the player when he is in space, planetUpdate - when on the planet. The planet calculates the acceleration and speed of the player. Depending on the horizontal acceleration, the angle of inclination of the plate changes from 30 degrees to -30. Approaching the barriers the player's speed decreases. Such barriers exist for the x axis (0; mapSize.x) and for the y axis. For the y axis everything is a bit more complicated. There is a minimum height, which is calculated as follows: the minimum height of the earth is taken, it is added to the height of the sine wave, and the height of the trees is still added. The height of the trees is calculated in a very simple way - the initial length of the branch is multiplied by the number of cycles performed when the tree is generated. There is no upper border - when flying out from the top of the map, the player switches to spaceUpdate and draws space.

SpaceUpdate works as follows: the acceleration and speed of the player is calculated. Next, calculate the angle of rotation of the player. The angle is calculated as follows: if the acceleration is zero, then the angle is calculated relative to the player‚Äôs speed, if not, relative to the acceleration. Also, in space, the player has the possibility of shooting. Shooting takes place as follows - a bullet is created with a turn as the player has and is added to the list. When a player is upgraded in space, each bullet in this list is also updated. When a player draws, bullets are also drawn. Also, in space everything is a little more difficult with barriers. Space is divided into sectors, in each sector 4 planets, in total - 1,000,000 planets and 25,000 sectors. Each sector has a unique id. If the remainder when divided by 500 equals 0, the left barrier is present, if the remainder 499 is right, if divided by 500, the result is 0, the upper barrier is present, if 499 is the upper one. If there are no barriers, then when flying out of bounds, the player moves to the appropriate sector.

Space


Most of the I have already stated, but still there are some things. In each of the sectors of space there are 4 planets. When a player presses the E key, if he is at a distance of a radius from this planet, then the player moves to the planet.

The enemies


AI enemies are very stupid - if there is a player in the radius of their visibility, then they simply tend to crash into it, and there is a small error, therefore their trajectory is quite a curve. If there is no player within the radius of their visibility, they are sent to their spawner.

Spawner


In each sector of space there is 1 spuner. Spawners can be of different sizes. Size affects the visibility of the player. If the player is in their zone of visibility, then the spawner creates enemies every 5 seconds, but the number of enemies cannot exceed 10.

Conclusion


After spending about a week, I created a game that does not use any .png or .jpg files.
