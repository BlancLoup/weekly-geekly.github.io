OAuth 2.0 and OpenID Connect support enhancements


OAuth 2.0 and OpenID Connect support has been updated and now includes the following features:


  • Support OpenID Connect Discovery 1.0 in accordance with the standard . During client configuration, the Management Portal now provides a new detection option that can be used instead of manually filling in information about the authorization server. The class % SYS.OAuth2.Registration now provides the Discover () method.
  • Support for dynamic client registration (OpenID Connect Dynamic Client Registration) in accordance with the draft standard . During client configuration, the Management Portal provides a new registration option that can be used instead of manually filling in client information on the authorization server. The class % SYS.OAuth2.Registration now provides the RegisterClient () method.
  • JSON Web Key (JWK) and JSON Web Key Sets (JWKS) support in accordance with the proposed standard for encrypting, decrypting, signing, and verifying JWTs signatures (JSON Web Tokens).
  • Since JWKS provides the means to represent a public key / private key pair; it is no longer necessary to use a certificate / private key pair for this purpose. The ability to use a certificate / private key pair is present for backward compatibility.
  • Rotation of keys (when using JWKS). The OAuth 2.0 configuration pages in the Management Portal provide options for key rotation.
  • Management Portal provides the ability to revoke tokens of specific users. This feature is provided for convenience and complements the existing client API.

All features supported in 2016.2 will work from 2017.1. For example, you may have a client and an authorization server running on different versions of Cach√©. However, new features are not supported by members working on version 2016.2.


In release 2017.1, changes were made to the configuration of the OAuth 2.0 and OpenID Connect classes. Cach√© will automatically update your saved configurations.

Mirroring enhancements


This release includes the following mirroring enhancements:


  • Significant performance improvements in the backup and asynchronous links of the mirror when recording updates (received from the main link of the mirror) to a local copy of the mirror logs. Demonstrated a 10-fold increase in throughput in the transfer of logs, which has expanded the scalability of mirror systems. As a result of these changes, systems using a backup mirror link for disaster recovery will also receive improved application performance and responsiveness of the main link.
  • The process of improving the status of asynchronous links for disaster recovery (Disaster Recovery) has been improved. When the link status in the emergency scenario is raised, before it becomes the main link, the link polls all the other links of the mirror in order to get the most relevant (at the time of the accident) log data from them. This ensures that disaster recovery includes all available data and other ‚Äúsurviving‚Äù asynchronous links to join the new main link. In addition, an intermediate step is maintained, allowing manual validation on a link whose status has been upgraded before allowing it to become a major one.
  • qcontrol list and ccontrol qlist now include the link type of the mirror and its status.

Feature tracker


In 2017.1, we present a safe and reliable approach to periodically gather information about the functions used by Cach√©, to maintain the integrity of relations with our customers.


Feature Tracker is controlled by a Cach√© task. By default, the task is suspended and the Feature Tracker is inactive, but it can easily be started by the user. Feature Tracker will help us prioritize which features of Cach√© to improve in the first place, based on what features our customers use.


Feature Tracker collects Cach√© instance attributes (product type, version, license, platform, etc.) and technology usage, including ECP, mirroring, encryption, and SQL. Feature Tracker does not collect information about license usage, database attributes, applications, errors, authentication, client data, and configurations.


Feature Tracker weekly sends the collected data (as an XML file) to a Cach√© instance on the InterSystems side using SSL. You can review the latest collected data to make sure that the process of collecting information is transparent. More information about Feature Tracker can be found in the Cach√© documentation in Feature Tracker Collects Usage Statistics .


Note: During the test period (Field Test), the Feature Tracker is active by default.
