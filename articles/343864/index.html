F # on Linux as a cure for the soul

Have you ever had the feeling that ‚Äúthis is‚Äù already tired? What do you want something new? "This" can be anything: a game, a job, a car. Anything that repeats every day. And in programming? Under the cut, you'll find a story about C # fatigue and choosing a more interesting approach.



I give the word to the author.


Recently, I was a little tired of C #, endless lines of the same type of code cease to warm the soul over time. At such moments, sometimes you want to do something for the soul. In my case, this is Linux and F #.


For the most part, it is important for me that I get out of my comfort zone. In general, I did, changed the system and language.


While organizing my vacation, I was faced with the fact that it was a little incomprehensible where to start. Let's see a little about how things are with F # on Linux.


Main:


  • Need lightweight editor with syntax support
  • Need support F # Interactive
  • Need a .NET environment
  • We need a debugger (as always)

Sophisticated workers have already realized that the " lightweight editor" + Microsoft = Visual Studio Code . I hope you already have it :)


Let's get down to business


So, with the choice of the editor figured out, now with everything else in order:


  • You need to install the mono and fsharp package, for most Linux systems you will find the instruction here . Although there are no Arch Linux instructions (and distributions based on it), these packages are present in the standard pacman repository, and the instructions show how to build from source, so there should be no problems.

In principle, in order to start something up, this will be enough, but then the article would quickly end.


  • Also put the .NET Core 2.0.0 (Runtime + SDK), again, for most systems, you will find instructions here . For Arch Linux, the easiest way to install the dotnet and dotnet-sdk-2.0 packages from AUR. If that, then here github After that, we will proceed directly to the preparation of VS Code for our work.

We put extensions to support F # syntax, build and manage Nuget packages.



For completeness, you can put the Nuget manager compatible with .Net Core.



We put the extension for debugging (Yes, everything is correct, C # is written).



After that, we are getting this rep , and then we follow the instructions: 


dotnet restore && code .

Now reboot the VS Code and wait until the extension of the "debugger" downloads its packages and will unfold normally.


In fact, now almost everything is ready, it remains only to test.


  • Create a folder somewhere, launch the console there. Next, perform: 

 dotnet new console -lang F# dotnet run

We receive treasured Hello World from F#!


  • Open VS Code in this code folder.
  • Go to Program.fs and select a piece of code and press Alt + Enter:


long-awaited F # Interactive starts.



It can also be used for more useful purposes:



Next, look at the build and debug.


  • To configure the build directive, press Ctrl + Shift + B and select .Net Core.


We do not need to change the automatically configured file.


  • After that, press Ctrl+Shift+B again and see that the build is successful:


  • After this F5, we also select .Net Core and then just change the path to the program.


  • Everything, after that you can put a breakpoint (F9) on our only line and check the debugger:


So, your car is set up and ready for new challenges in a wonderful functional language. A nice bonus is that your work can be built into the C # project (by connecting. Dll).
