Maximum performance


Almost all reports deal with performance in one way or another, even if in passing. This is a feature of the developer. But some want to highlight specifically, these are the reports of four speakers:

  • Konrad kokosa
  • Yegor Bogatov
  • Evgeny Peshkov

We have already seen a report about our own .NET runtime. What about your own GC? Will we need to patch a horrible file the size of two megabytes of generic C ++ code? Fortunately, no. In .NET Core 2.1, a new feature has been added called Local GC, which allows you to completely replace the standard garbage collector with something handwritten or, on the contrary, use the standard collector outside the usual .NET environment. In the "Make your custom .NET GC -" whys "and" hows "report, Konrad Kokosa will introduce us to how this is all done. It will be interesting and useful mainly for those who want to get deeper into the memory management and behavior of GC. The main point of the report is not so much the fact that you came home from the conference and immediately transferred the product to a self-written something. Rather, it‚Äôs a cool inspirational report that broadens the boundaries of the possible. You do not have to do this, but now you have the opportunity to experiment. GC seems less and less an incomprehensible thing in itself and more and more falls into the hand as a convenient and obedient tool.



But GC is just the beginning. Recently, in many runtimes it has become popular to give the user access before generating processor SIMD instructions, such as SSE and AVX. For example, in JS, they added (and removed) SIMD.js, in Java, the Panama project deals with this, and so on. Not bypassed this mod and .NET. We were given super powers, but they are such that it is not enough to have them - you need to be able to use them, and this is difficult. On the one hand, control at this level requires tremendous motivation and qualification: to write a vector code that surpasses the scalar performance, you need to tense up not only with things like alignment, but also follow specific instruction sets, think about the result of generation and things that happen if you go off the fast track. On the other hand, if you really write the maximum performance application and save every beat, then you can meet with the stupidity of even the most advanced compilers. Even such a comparatively studied problem, the allocation of registers, is NP-complete ( one , two ), requiring human intervention, and it is unlikely that the situation with SIMD was something better.
