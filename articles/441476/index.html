Making the JaCarta Editor

‚ÄúI can say only one thing about the Kingdom of Shadows: there is reality and there is its Shadow; that's the essence of everything. In the real world, there is only Amber, the real city on the real Earth, in which everything is collected. And the Kingdom of Shadows is just infinity of irreality. It all started with the ability to work with the same key on etoken from different work places that are significantly distant from each other (USB Over IP for a couple of tokens will be expensive) and my great desire to open this closed world. I came across the work habr.com/post/276057 for which its author has a lot of respect, in my project a significant part of the functions it has been used is used (the code is open). True, as it turned out, everything that works with etsdk.dll works only with blue fishes. Therefore, for JaCarta, new functions had to be written anew, and some were edited.

As a result of lengthy research, the JaCarta Editor appeared - a program that shows and allows editing entities (the so-called token file system objects are in official documentation, apparently hinting at their ephemeral and unreality) on Aladdin's tokens, including the most modern ones.

A similar application for Rutoken is in the public domain as part of the Rutoken SDK (Rutoken Editor), but for Aladdin, at least not in the public domain, although 15 years ago, judging by the documentation that was found on the Internet, this was (ETEditor).

The program is written in Autoit, tested with EToken PRO Java 72 K, JaCarta LT, JaCarta Pro, JaCarta GOST-2.

The program will not work if the Rutoken drivers are installed on the computer (it is required that EToken or Ja Carta have reader number 0).

For the same reason, only one token must be connected for correct operation.

The script uses the Window $ system calls, and has been tested only with this OS family.

To work, you need to install the "Single client JaCarta" (downloaded for free from the manufacturer's website), when installed, a much newer version of etsdk.dll is installed in the windows system folder, as well as jcFS.dll containing JaCarta File System functions (very similar to those that are in etsdk.dll, but jcFS has significantly more functions without which working with some modern jacards, for example, GOST-2, will not be possible). If the Single Client is installed, it is of course not necessary to search and store this dll in the program folder; otherwise, when the specific token type driver is installed, this dll should be in the program folder.

All jcFS functions are described in detail on the manufacturer‚Äôs website, but nevertheless some parameters of these functions had to be selected experimentally.

In order to get access to the root directory of the token, you must call the ETRootDirOpen or JCRootDirOpen function (which is the same, since the first in the dll calls the second, this is true for almost all functions, but there are a few exceptions) with the vendor ID equal to 0 (constants like 0xF007 will allow see only individual directories, mostly old eToken, which store bank keys generated using PKIAdmin and MessagePRO utilities).

The next important feature of modern jakarts is the need to know the id of the applet that is installed on the token, the JCSelectApplet function is responsible for this.

Older tokens usually have a default applet and the call to this function is not specifically required, but for new and relatively new jakarts without knowing the applet's id, further work becomes impossible. Calling all the functions that receive information from the token in this case returns False. In my project, in this case, a two-stage brute force is implemented (the token always responds to two id, but only one of them works) id and further work with the desired value.

All information is displayed in the output window located under the directory tree.
The input of the PIN code is implemented, the information about the remaining attempts to enter it is also displayed.
The contents of the directories and brief information about the files are displayed: the name, whether the file is private information, pin-private code - the letter p: (zero after it is public, the unit is private) and the file size in bytes after ‚Äús:‚Äù. The file is opened by double clicking.

You can view and, if necessary, copy the contents of the token files to a buffer in hexadecimal or save them to a computer in binary form.

You can also change the contents of the file and select ‚ÄúSave changes‚Äù in the menu (a pin code must be entered first, if not entered, a corresponding message will be displayed, in this case the edited data can be selected and copied to the clipboard).

To delete a file, select it and click "Delete", after which the confirmation window will appear.

JaCarta File System has an interesting feature that is present on all the tokens that I tested. If you create a directory or several directories without files on the token, they will disappear in the next session of working with the token, apparently this way the file system takes care of saving the token‚Äôs memory and cleans it of any garbage.

Therefore, when you click the Create button in the program, a chain is immediately created from a directory or two directories and a file. The nesting depth of directories in the program is two, not counting the root. You can create only directories in the root directory, but not files.
Before creating, editing or deleting entities, you must enter the pin code.

Very important note about directory names!

On tokens, where content is created by the manufacturer, you will never find directories with the same name, no matter where they are. This is due to the features of the file system.

Suppose we have the following entity on the token: // 0001 / A001 / 0008 (that is, the folder 0001 is in the root directory, the folder is A001, and 0008 is in it) and we create a new entity on the token: // CC00 / 0001/1010 . When accessing the file 1010, the search of the directory 0001 will start, which is in the root and will be found first, but there is no such file in it. As a result, the function returns False and the essence becomes lost, as it is impossible either to access it and delete it. Only token initialization will help.

Particular noticeable about EToken PRO and JA Carta Pro, the difference in which only in appearance, they have system files in the root directory (they are shown on the screenshot), one of which is stored, apparently, hash pin code, changing these files leads to the fact that authorization becomes impossible (the pin code becomes incorrect) and only the initialization will help after that.

Regarding the private keys, it was possible to establish the following: if a container created by a third-party cryptographic provider, such as CryptoPro, is stored on the token, all its contents, including private keys, will be available, it‚Äôs just a flash drive with a pin code (as shown in the picture). If the key pair was generated on board the token using PKCS11, then only the certificate and the public key will be available.

Since this is an experimental program, I do not recommend using it with tokens containing valid EDS keys, all operations are performed at the peril of the experimenter. This especially concerns the creation of new entities, in some cases the result is difficult to predict.

All information for creating the program was obtained from open sources, the purpose of its creation is purely research.
