User Account Control (UAC) bypassing by spoofing trusted directories


Information security expert David Wells has published a way to bypass UAC account control in Windows 10

Hello!

While researching some of the new UAC circumvention techniques (UAC), I discovered a completely new UAC circumvention method at the time of this writing. It is worth noting that Microsoft does not consider UAC a security boundary, however, we still report different bugs at Microsoft and I want to share the details of the vulnerability I found here. This method was successfully tested on Windows 10 Build 17134. Before I dive into the details of my find, I will first give you a little explanation on how the UAC service works.

Uac primer

When a user in the Administrators group wants to start an application that requires elevated privileges, the UAC displays the corresponding request and the user who is a member of the Administrators group will need to confirm the action. However, this UAC request does not occur for ALL administratively executable files in Windows . There are a few exceptions that will ‚Äúautomatically‚Äù elevate the privileges of an executable file without causing a UAC request, bypassing UAC (much to the surprise!). This particular group of select trusted executables passes additional system security checks to make sure that these files are in fact reliable, so information attackers usually do not abuse this function. This approach has been used in previous UAC workarounds and will be the basis of my new workaround. However, there are a few loopholes that we need to use in order for our attack to succeed. Let's look at the requirements that must be met if we want our executable file to be ‚Äúautomatically elevated in privileges‚Äù. To do this, I will show a few pictures of the appinfo.dll disassembled library (the AIS service that processes requests for privilege escalation is one of the main components of UAC).

Requirement 1: The file is configured for automatic privilege escalation.

When a request for elevation of privileges for a program occurs, an AIS call (appinfo.dll) makes an RPC call with the target executable path passed as an argument. This service will then match the target executable contents of the file for reading. In the manifest of the executable file, an attempt is made to read the value to obtain the key ‚ÄúautoElevate‚Äù (if it exists).

UAC bypass

As mentioned earlier in the UAC Primer section, autoprivilection (UAC bypass) will be performed for executable files that:

  1. Configured to receive ‚Äúautomatic‚Äù privilege escalation
  2. Correctly signed
  3. Runs from a trusted directory ("C: \ Windows \ System32")

Appinfo.dll (AIS) uses the RtlPrefixUnicodeString API to check if the executable path of the file with "C: \ Windows \ System32 \" matches one of the trusted directories. This is a fairly reinforced concrete check, considering its comparison with the canonical location of the file.

Therefore, to organize a crawl of this check, I create a directory called ‚ÄúC: \ Windows \‚Äù (note the space after ‚ÄúWindows‚Äù). Of course, using this action still cannot pass the RtlPrefixUnicodeString check, and I will also mention that this is a somewhat invalid (or at least ‚Äúunfriendly‚Äù) directory name, because Windows does not allow spaces at the end of the name when creating the directory (try ).

However, using the CreateDirectory API and adding ‚Äú\\? \ ‚ÄùTo the name of the directory that I want to create, we can bypass some of these name filtering rules and send a request to create a directory directly to the file system.



This leads to the creation of an inconvenient directory that happily coexists in the file system along with the real "C: \ Windows \" (unless you try to do something with it in Windows Explorer).



Now that we have the ‚ÄúC: \ Windows \‚Äù directory, we can create the ‚ÄúSystem32‚Äù directory in it and copy one of the signed, executable files (which are allowed by the system for ‚Äúautomatic‚Äù privilege escalation) from the real directory ‚ÄúC: \ Windows \ System32 ".
To do this, I copied "winSAT.exe" (one of the files in the whitelist of Windows executable files with the system enabled "automatic" privilege elevation).
When we try to run this file from our new directory ‚ÄúC: \ Windows \ System32 \ winSAT.exe‚Äù, it will go through the following APIs (see Figure 6) in appinfo.dll before performing the verification of the trusted directory. This is important, and the basis of why this bypass works.

When this awkward path with a space is sent to AIS for a request for privilege escalation, the path is sent to GetLongPathNameW, which converts it back to ‚ÄúC: \ Windows \ System32 \ winSAT.exe‚Äù (the space is removed).

Now this is the string in which the valid directory was validated (using RtlPrefixUnicodeString) for the rest of the validation.

The beauty of my solution is that after checking the trusted directory, this converted path is executed, which is then released, and the remaining checks (and the final request for privilege escalation) are performed with the original name of the executable directory (with an extra space).

This allows all other checks to pass and causes appinfo.dll to accept my copy of winSAT.exe as with ‚Äúautomatic‚Äù privilege escalation (since it is correctly signed and added to the white list for ‚Äúautomatic‚Äù privilege escalation).
