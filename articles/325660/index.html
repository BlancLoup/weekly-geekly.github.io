The State of Reactive in JS: A Practical Review of FRP Libraries

The article was born from this response to StackOverflow and developed into a blog post ( in English ), but I think it has not lost its relevance.

Now a lot of noise around Reactive Programming. I spent some time studying this discipline applicable to JavaScript and Node.js and put my opinion on the most interesting libraries in my opinion.

Rxjs


This is a kind of benchmark - the emergence of Reactive Extensions for .net, in a sense, created this discipline.

Pros:


  • Documentation and a large community ( > 15,000 stars on Github , > 3073 SO tags );
  • This is a library from the Rx * family, which means it uses the same interface. Practically, this means that sometimes you can watch examples from RxJava and the original Reactive Extensions;
  • There are primitives for almost all occasions;
  • There is a backpressure ( here you can read what it is );
  • There is one book for this library.

Cons:


  • Sometimes it looks like "matan". Documentation resembles an abyss - it is not clear how to pull it all over your head?
  • Implemented by guys with Java in their head (love for a large number of abstractions, idiomatics from Java / .Net, sometimes strange signatures). It's not that bad, just some things look strange in JS ( Scheduler example - I still don‚Äôt understand why all this is necessary in a single-thread environment). Hence the desire to write on TypeScript

Bacon.js


Once a fashionable library for hipsters - now everyone has forgotten

Pros


  • Good documentation and good examples from the community (for example, Snake game );
  • Almost all primitives from Rx * are implemented;
  • Implemented at first very for JS and JS with the background;
  • IMHO, it is very easy to start just walking through examples and documentation.

Cons


  • No backpressure - let's just forget about backend development
