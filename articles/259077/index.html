How encryption works


Timur has a suitcase and a key. He passes the empty suitcase to his friend Sasha.


Sasha also has her own suitcase and key. He makes a copy of them, and hides Timur in his suitcase.


Since the suitcase is returned back closed, only Timur can open it. Only Timur has the key to his suitcase.


Timur opens his suitcase and takes out a copy of the suitcase and Sasha's key.


Now Timur no longer needs his suitcase and his key. He can use those that Sasha gave him.


This is how the RSA encryption algorithm works. And this is how a fast decryption of a crypto-resistant algorithm using a thermorectal cryptanalyzer looks like.


The technical side of the issue


I chose RSA as the encryption algorithm. Despite the fact that the algorithm is about 30 years old, the first libraries found on JS turned out to be of extremely poor quality. It seems that they were written exclusively for one project. With difficulty and pain I was able to adapt them to the needs of my project, although the tasks were typical:
  • Generate keys
  • Encrypt text with public key
  • Decrypt text with a private key


I was also surprised that in half of similar extensions, you need to invent some phrases, somehow transmit them, etc. Although, it seems like a long time ago people transmit encryption through open channels, and the process is amenable to automation.

Problems parsing social. nets


Started with VKontakte. The first difficulty is that each dialog has its own input field and message display field. The problem was solved by obtaining the id of the interlocutor.

Next, it was necessary to intercept the ‚ÄúSend‚Äù button and the Enter button. The problem is that VKontakte first hangs up events, and only then the plugin is initialized. To fix the situation, I added my own handlers to document.body, and after that I filtered all the intercepted events. It was necessary to receive them first in order to have time to replace the original message with encrypted.

The site classmates showed that this is not enough. Their maximum message length is 3000 characters, and the encrypted key pair has a length of 3800. In addition, regular messages also become longer by converting to base64. I had to write a mechanism that would split the messages into parts, sign them (id, current index, total number of parts), and then collect them back. It turned out something like packets in TCP / IP.

Next, it was necessary to learn how to generate the ‚ÄúSend Message‚Äù event, since the script needed to send packets by itself. In VKontakte, the problem was solved very simply, by calling the global method IM.send (). To get to the site scope from the extension scope, I had to add the SCRIPT tag and write a method call to the line. Example: 
<script>IM.send()</script>

Here it should be clarified that the DOM on the site and the extension are the same, and the global scope is different. In classmates I could not find a similar method. Guys minimize code and do not produce globals. I had to emulate an event on the "Send" button: 
 var event = new Event("click", { bubbles: true, cancelable: true }); button.dispatchEvent(event);

When the text was encrypted, divided into parts, sent and assembled, it was necessary to distinguish between new and old messages that appear on the site. Both on VKontakte and classmates they are signed by a date or a unique id ... I thought ... I have never been wrong. After this, a ‚Äúfloating‚Äù bug appeared. It turned out that all these tags can, in one way or another, be the same for two different messages. Therefore, I began to assign my attribute to all DOM nodes in order to know who was already processed and who was not.

But the miracles did not end there. Classmates published a message, then I replaced it decrypted, then some kind of magic happened, and after a few milliseconds classmates again wrote the original text the same HTML element. I had to make a clone of the message nodes, write a decryption there, and hide the original element. The original node was hidden (display: none), but was not deleted, because I did not know how and why they rewrite it. There was a danger of breaking the classmates script if the HTML element was deleted.

The last riddle of parsing social. networks consisted in the fact that if several people logged in from one computer, then their pairs of keys for the dialogues needed to be saved in different places. The problem was solved by retrieving the current user ID. Knowing the user ID made the ability to create personal records in localStorage: 
 rsa_keys__326374454 rsa_keys__43234223 rsa_keys__113234753 ...
