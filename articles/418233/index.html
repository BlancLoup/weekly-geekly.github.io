Linux, Lucene, Mmap

  • redhat-6 - /etc/security/limits.conf

  • create a database connection;
  • read file;
  • write to file.
  • ...

If your system is actively working with files and doing a lot of operations, the setting should be increased. Otherwise, the slightest load will put you.

You need to know about this and immediately write a manual: configure the operating system on the server, otherwise there will be such and such consequences. And on its side it is important to carry out not only functional tests, but also stress, on the actual amount of data.

Oracle RAC


Oracle is a popular database. Oracle RAC is when you have several database instances. It is necessary to ensure the smooth operation of business-critical systems: even if one instance is broken, the rest continue to work, the user will not even know about it.

If you are using Oracle RAC, you MUST have load testing on it. If you do not have it, then you need to ask the customer, with whom he stands, to carry out the load on his side.

Then the question may arise - why then do you not have it? It's simple, iron for testing usually always give worse. And if the system is focused simply on Oracle and one customer out of twenty uses RAC, then buying it for testing will be unprofitable, since RAC is very expensive. It's easier to negotiate with the client and help him conduct tests.

What happens if stress testing is not performed? Here is an example from life.

There is an opportunity to create a column in the database and say that it is an auto-incremental field. This means that you do not fill out the field at all, it is generated by the database. Is there a new line? Recorded the value "1". Still new line? She will have the value "2". And each new value will be more and more.

For example, it is very convenient to generate identifiers. You always know that your id is unique. And for each new line it is more than for the previous one. In theory ...

We have two entity identifiers in the system:

  • id - identifier of a specific version, auto-increment field;
  • hid is a historical identifier, it is always constant for one entity and does not change.

As a result, you can make a selection for a specific version, or you can make a selection by a hid-entity and see its entire history.

When an entity is created, id = hid. And then the id increases, it is always greater for new versions than hid. Therefore, the formula for determining the version:

version = (id - hid) + 1

It cannot be negative, since the base itself creates the id.

But they come to us with a question on the essence and show the records from the database. I don‚Äôt remember what the question was about, and that‚Äôs not important. I look at the records and do not believe my eyes: the version has negative meanings. How so ?? This is impossible. It turned out possible.

In the RAC, each node has its own cache. And it may happen that the nodes do not have time to notify each other, and you see the same number twice in the table:

  • . , ? , 10. , 11.
  • ( 1, 2).
  • , ? , 10 ( , ). , 11.



Total we receive not unique value of a unique field. And in fact, with a large load of such intersections of identifiers there will be not one or even two ... If you have all the business logic tied to the fact that id is always unique and always increases, there will be OY.

In our case, nothing catastrophic happened, and carrying out load testing on the test bench of the customer helped. Found the problem early, as it turned out, negative versions of the system do not interfere with life. But we have added to the scripts for creating the database the ordering of sequences, just for such cases.

The moral of this story is that it is VERY important to carry out load testing on the same hardware that will be in PROD. Everything can affect the result: the settings of the OS itself, the settings of the database, whatever. What you did not even suspect.

Perform tests in advance. And remember that not all problems can be found functional tests. In this example, a simple test would not find bugs. After all, if you create entities manually, that is, slowly, all the nodes of the database will have time to notify the neighboring ones, so we will not get inconsistency.
