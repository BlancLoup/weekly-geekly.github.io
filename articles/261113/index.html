K-Meleon 75: Reached the final

So, it happened: our favorite browser was released in the final version of the 75th version.

Six months of work on this very large-scale step in the development of the product ended with the release of the assembly, which one of the old and very advanced users has already managed to call the most stable core of all recent builds.

Actually, what should be the release.

The full list of innovations, however, looks both impressive and relatively modest at the same time.


The thing is that the 75th version was largely devoted to changes "under the hood", the most ambitious since the time of version 1.5 - well familiar with the history of the browser perfectly understand what it says.

For the first time ever since (release 1.5 took place in the summer of 2008) one of the key components of K-Meleon's uniqueness, the macro language , was seriously updated. 20 new functions were added to it at once (this will require a separate conversation, but I still have to prepare for it myself).
The functionality of the browser interface with javascript tools has been seriously upgraded and extended through the JSbridge module (here I separately invite not only and not so much to the discussion as to the complicity of JS developers - my qualification, alas, is not enough for not only use, but also for conscious descriptions of this functional.) In any case, we can say that Kameleon makes another step towards extensibility, additional third-party functionality and integration.

Now we need people who will actively use all this good, bug report, fix and so on.

Another, almost revolutionary, change was the ‚Äúreform‚Äù of the browser's skinning system , although here, almost all the changes occurred ‚Äúinside‚Äù and are mainly interesting to those who are going to start working on creating program interface options. In general, now it will be much easier and faster to collect visual themes, thanks to the full support of the PNG format with true transparency, rather than its imitation with the help of the ‚Äúmagic vanishing pink‚Äù.
For the user, only the ability to quickly (via GUI settings) change the thickness (height) of toolbars was added immediately. This can be viewed as an adaptation not only to the different requirements of users, but also to the new ‚Äúiron‚Äù reality with high and ultra-high resolution screens, as well as with touch interfaces (it is clear that it is much more convenient to poke fingers into large buttons).


All the changes described today look more like investments in the future: they need developers and advanced users who will use all this. It is important that K-Meleon is confidently fixed in the status of not just a browser, but a scalable (geometrically and functionally) platform that can be flexibly adapted to different environments and needs.
And before, the browser could, if desired, be turned into almost a full-fledged basic working shell, and now even more so.

Of the remaining innovations, the most important for the user is, perhaps, the integration of various Firefox tools , which makes the cross-browser experience much more consistent and compatible: here and gradual migration towards the native for the new Firewall interface settings through about: preferences, and support for the download manager (which was never before in KM), and, of course, the greatest possible at the present stage of overcoming architectural differences, allowing Chameleon to use some (not so small) part of the Big Wall extensions ta fox. (At this point, I again call into our ranks experienced developers, because it is possible - and necessary, as I see it - there is still a lot to improve and add, but professionals of all kinds of coding are required).

In addition, the own functionality has been seriously refined - in particular, the sessions module got rid of some annoying vintage glitches, learned to remember the last active tab, and the architectural sessions are finally allocated into a separate json file, which can also be seen as a contribution to future convenience of working with sets open pages.

The spelling checker module, dropped from the past, 74th release, was re-implemented in the browser.

For the rest, there was an unequal struggle with the old and newly manifested glitches (crashes, resource leaks, improper functioning), and many of them were defeated.
The 75th, of course, is not 100% sinless, but we can safely say that it is better than the 74th, not only by the breadth of functionality, but also by the stability and quality of implementation of what users have been used to for a long time.

The release will not become a point of frozen narcissism: first, it has already revealed some annoying (but fortunately, minor) bugs; secondly, Dorian and the community already now have a certain amount of accumulated material for a minor update.
