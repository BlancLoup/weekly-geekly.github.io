Developing a VPN Client for Android (Part 1)

Hello! The reason for writing this article was the awareness of the fact that when there are a large number of articles and reviews about VPN application clients for Android, there is not a single normal article describing development problems using the VpnService API. Moreover, in most cases you, as an application developer, cannot do anything with these problems.

Let's start from the beginning. Some time ago, on the basis of research conducted in the field of security of mobile devices, our company decided to release a small application ( WebGuard ) under Android to block all already annoying ads, as well as protection from surveillance and viruses when working from any browser. To implement this functionality, we needed to solve many problems, the most time consuming of which was the task of intercepting and processing application traffic. To intercept and filter browser connections, it was decided to use the VpnService API, which appeared in Android from version 4.0.3 and provides all the necessary functionality (though at times this functionality simply does not work for a bunch of different reasons, but it turned out a little later).

A little about the technologies used


Here it is worthwhile to tell in more detail why this API was chosen and what methods exist at all for intercepting network traffic of applications by another application on a ‚Äúnon-routed‚Äù android device. Actually, there are only three methods (apart from various vulnerabilities) and it will be useful for application developers with various mechanisms of in-game purchases to know about them. Because many of them believe that the substitution of network traffic for an application is possible only on the ‚Äúroot‚Äù device and does not bother with the protection of transmitted data, although there is a possibility (and quite large) of the appearance of ‚Äúcheat‚Äù applications that make changes to the transmitted data.

So, the first way is to install a local proxy server . Most antiviruses for Android use this method. An application for intercepting traffic just needs to implement the HTTP proxy support described in RFC 2068 and set the proxy server or APN in the settings of the mobile network in the WiFi network settings. In this case, the data transfer will occur as in the diagram below.



But this method has a lot of serious problems:

  • You can programmatically install a WiFi proxy or APN for a mobile network only using a hidden API (for example, hidden methods in WifiManager);

  • you need to monitor the emergence of new WiFi or mobile networks by setting up a proxy (APN) for them;

  • This method works only for connections created using the Http (s) URLConnection class and, unless explicitly stated, do not use a proxy (for example, url.openConnection (Proxy.NO_PROXY)). There are also applications that check the presence of a proxy server and use it (or vice versa do not use it);

  • if the user deletes or simply does not start the application with the proxy server, other applications will not be able to connect to any server, because the network settings will still have the proxy specified and the user will have to remove it from the settings ( for example );

  • it is impossible to build a chain of several proxies, respectively, and there can be only one intercepting traffic application.

All these problems, except for the last, are successfully solved by the second method - by writing a VPN client application using the VpnService and VpnService.Builder classes. It is enough for an application to call a couple of functions (mandatory checks are omitted in the code):

"Features" VpnService API


In the process of developing the application, testing and using the first versions by users, we are faced with a large number of errors or flaws associated with the VpnService API. Some of them managed to be fixed, since in fact these were flaws of our programmers (which we honestly wrote and got on bash.org.ru), and with the rest of it, it is difficult or impossible to do something:

  • There is no support for the TUN interface in the linux kernel , so VPN will not work. Basically, this problem is encountered on self-assembly firmware based on CyanogenMod, AOSP, and other projects. Assembly authors either remove support from the kernel altogether, or forget to put the tun.ko module in the firmware. It seems that the authors are guided by the principle ‚ÄúI don‚Äôt know what it is, so it‚Äôs not necessary‚Äù;

  • There is no file VpnDialogs.apk which contains the dialogs of the request for rights and statistics, VPN again will not work. Strangely enough, this error most often happens on official assemblies from phone manufacturers ( including Google);

  • GUI freeze on some Samsung phones with official firmware, if an unhandled exception occurs in the VPN client (and the shell via the ADB is quite working, but reconnects every 1-2 minutes). In general, manufacturers of phones with Android, sometimes collect ‚Äúgreat‚Äù firmware for them, forgetting to check out some of the ‚Äúrarely used‚Äù functionality. Our development team in this regard simply "adores" Samsung and believes that to test Android applications, it is imperative to have several phones from different manufacturers and several different phones from Samsung;

  • Thanks to this change, the user will not be able to tick ‚ÄúI trust this application‚Äù in the VPN dialog if it is blocked by another activity that skips user clicks. This is done so that it would not be possible to draw on top of another dialogue and trick the user into putting a tick. The problem is that various utilities, launchers and some programs (for example, ‚ÄúRead! Free‚Äù) usually do this (and for a long time), creating a transparent activity. As a result, some users might think that they are being bullied;

  • If the user turns on traffic restriction for background processes, then a working VPN client will be ‚Äúdistributed‚Äù (when sending data via a socket, there will be a SecurityException exception with Permission denied), respectively, there will be no Internet kin ;

  • Linux has such a wonderful mechanism as OOM Killer , and if the user suddenly launches an application that will ‚Äúdevour‚Äù a large amount of the device‚Äôs RAM (for experiments, you can take Firefox, that's why there is always little memory), OOM Killer will start to ‚Äúdump‚Äù everything consecutive user processes except, as usual, the hero of the occasion (more precisely, he will be killed last). There are no exceptions for the necessary processes (launcher or VPN client), so you need to be ready to constantly restart your service;

  • After an incorrect shutdown of the VPN client (even if it restarts afterwards), in rare cases, you may need to reboot the device, because Android will incorrectly reconfigure packet routes and packages from applications will go to / dev / null ;

  • If the user turns on the distribution of the mobile Internet via WiFi simultaneously with the VPN client, then on most firmware, as in the case above, Android will incorrectly configure packet routing;

  • Despite the decent list of features, Google developers may suddenly decide to remake the improved functionality in the new version, and Android developers usually do not warn about all changes, so you need to carefully monitor your Android committees and edit your code. The result of this approach is most often felt by the users of Nexus phones, who receive the update the fastest (read heartbreaking requests in the style of ‚ÄúPLIZZZZZZZZZ !!!!!!!!!!!!
