About genealogical programs.



Why is it needed?

1. Storage of information. If everyone
stored on paper - may burn, divide, etc. In electronic form if you regularly backup less
2. Organizing information. Quickly find all the people who died in the war. all second cousins ‚Äã‚Äãof this person, all who were born in Kiev before the revolution. Such tasks often arise during work. To get all such slices is very difficult if everything is stored in one file or on paper.
3. Work with sources. It is often important to know from which source the information came. Often it is necessary to find everything that is known from this source.
4. Work with photos. Store all pictures related to one person together. If there are several people in the photo, do not store it several times Attach photos not only to people but also to specific events.
5. Work with geographic places. Description, photos, previous name, a list of all who were born there.

Now let's talk about GEDCOM - the format of genealogical data.

GEDCOM

Each genealogical program has its own data storage format.
Therefore, to transfer data from one program to another, in order to send someone their own tree or subtree, use the standard format - GEDCOM.

The format is text. Current standard 5.5
Describes a hierarchical data structure (like XML). Person section, family section, source, multimedia ...

In all normal programs there is an import / export in GEDCOM - do it, open it in a text editor - look.
It is instructive :)

What are the pitfalls?

  1. set of tags
    (i.e. event types, properties) in the standard is limited. For example - not in
    standard nothing describing army service or participation in war. therefore
    Each program adds its own tags. And, as a rule, adding an event to
    a description of a person or a field about him (an alternative name, for example), you do not
    you know - this field is from the standard or not. In addition, many programs
    allow you to add your own types of events. As soon as you exported your
    database in GEDCOM and trying to import it into another program - that
    starts shouting: "I don‚Äôt know such an event!"
    There are different options - the program will make all unfamiliar events notes (Notes) - you are in the import log all
    you will find them and do something with them (leave in notes, replace with
    standard tags), the program makes all unfamiliar events custom events, you see which types of events are suitable for you in this program and do an import / search in the GEDCOM file before importing
  2. Encoding.
    If you have sources or, even worse, names in Russian or Hebrew, then
    so that the program can correctly export non-Latin encoding skillfully
    import it correctly.
    And there may play a role which on a specific computer default codepage (default encoding). In general, I would advise, if it is supposed to exchange data with foreign countries, the names are initially entered in Latin letters. And maybe the names of cities. And then at the request of a relative in America to send a tree, you will be scratching a turnip for a long time.
  3. Ways to pictures and other multimedia objects. The path can be stored in
    the program (and recorded in the GEDCOM file) either absolute or relative.
    If the path is recorded as absolute, even when transferring the database from one
    computer on another, on the same program, there may be problems


In short - the conclusions:

  • If you need to share your tree with someone, use Export to GEDCOM
  • Without a special need not to switch from program to program, because the transfer
    A large database can be quite painful.
