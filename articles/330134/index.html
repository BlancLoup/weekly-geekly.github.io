Synchronize database structure between applications

Anyone who has ever developed applications that use the database, probably faced with the problem of updating the database structure when deploying and updating the application.

The most common approach is to create a set of SQL scripts to modify the database structure from version to version. Of course, there is such a powerful tool as Red Gate , but firstly it is not free, and secondly it does not solve the problem of full automation of the update.


The migrations technology, first introduced in ORM Hibernate and implemented in Linq, is very good and convenient, but it involves the code first structure development strategy, which is very time-consuming for already existing projects, and using triggers, stored procedures and functions in the database makes the transition to code first virtually impracticable.


This article proposes an alternative approach to solving this problem, using the storage of the reference database structure in the XML file and automatic generation of the SQL script based on a comparison of the reference and the existing structure. So, let's begin...

Deploy / update the database structure using the resulting XML.


Now learn how to use the resulting XML. Create another empty DbSyncSampleCopy database, add the following to the code of our console program: 


  class Program { private const string OrigConnString = "data source=.;initial catalog=DbSyncSample;integrated security=True;MultipleActiveResultSets=True;App=EntityFramework"; private const string TargetConnString = "data source=.;initial catalog=DbSyncSampleCopy;integrated security=True;MultipleActiveResultSets=True;App=EntityFramework"; static void Main(string[] args) { //  XML     var dborig = new Shed.DbSync.DataBase(OrigConnString); var xml = dborig.GetXml(); File.WriteAllText("DbStructure.xml", xml); //       ,  // Shed.DbSync.DataBase.ClearDb(TargetConnString); //     var dbcopy = Shed.DbSync.DataBase.CreateFromXml(xml); dbcopy.UpdateDb(TargetConnString); //       : // dborig.UpdateDb(TargetConnString); // dbcopy         XML } }

After starting the program, you can make sure that the structure of the tables, identical to the reference database, appeared in DbSyncSampleCopy. Experiments with changing the reference structure and updating the target leave the reader.


In testing scenarios, you may need to create a test database every time from scratch. In this case it will be useful to use the function Shed.DbSync.DataBase.ClearDb (string connString)


Automatic tracking of the structure of the database.


Tracking the structure will be issued as a separate function, which should be called at the start / restart of the application or in another place at the request of the developer. 


  static void SyncDb() { //      Shed.DbSync.DataBase.Syncronize(OrigConnString, @"Struct\DbStructure.xml", //     @"Struct\Logs", //      @"Struct\update_script.sql" // (.)      //     ,  //    ); }

Tracking is performed using the Version (tag) parameter in XML. The scenario of using the procedure is as follows:

  1. Assign DB version. In Microsoft SqlServer Management Studio on the required database node, right-click Properties.
  2. Next, Extended Properties and in the property table add the Version property with the value 1. With each subsequent modification of the structure, this property should be increased by 1.
  3. When launching an application, if there is no XML file or its version is smaller than that of a database, it is created.
  4. If the version of the XML file is larger than that of the database, a script to update the database is generated and executed.
  5. If errors occur during the execution of the script, all changes are rolled back.
  6. The synchronization results are written to a log file created in the folder specified by the logDitPath parameter.
  7. If the SqlScriptPath parameter is specified, a file is created with the script in Section 4.
