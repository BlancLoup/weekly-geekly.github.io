Initializing the page directory


Our kmain method took three arguments as input, one of which is the virtual address of the page table. To use it later in the allocation and memory management, you need to designate the structure of records and directories. For x86, the Page directory and Page table are described quite well, so I‚Äôll confine myself to a small intro. Entry Page directory - structure by pointer size, for us - 4 bytes. The value contains the physical address of the page aligned to 4 KB. The lowest byte of the record is reserved for flags. The mechanism for converting a virtual address into a physical one looks like this (in the case of my granularity, a 4 MB shift occurs by 22 bits. For other granularities, the shift will be different and hierarchical tables will be used!):


Virtual address 0xC010A110 -> Get the index in the directory by moving the address 22 bits to the right -> index 0x300 -> Get the physical address of the page at index 0x300, check the flags and status -> 0x1000000 -> Take the lower 22 bits of the virtual address as offset, add to the physical address of the page -> 0x1000000 + 0x10A110 = physical address in the memory 0x110A110

To speed up access, the processor uses TLB - translation lookaside buffer, which caches page addresses.


So, this is how my directory and its entries are described, and that same setup_pd method is implemented. A ‚Äúconstructor‚Äù method is implemented for page writing, which guarantees alignment by 4 KB and setting flags, and a method for obtaining the physical address of the page. The directory is just an array of 1024 four-byte entries. The directory is able to establish the association of a virtual address with the page using the set_by_addr method. 


 #[derive(Debug, Clone, Copy, PartialEq, Eq)] pub struct PDirectoryEntry(u32); impl PDirectoryEntry { pub fn by_phys_address(address: usize, flags: PDEntryFlags) -> Self { PDirectoryEntry((address as u32) & ADDRESS_MASK | flags.bits()) } pub fn flags(&self) -> PDEntryFlags { PDEntryFlags::from_bits_truncate(self.0) } pub fn phys_address(&self) -> u32 { self.0 & ADDRESS_MASK } pub fn dbg(&self) -> u32 { self.0 } } pub struct PDirectory { entries: [PDirectoryEntry; 1024] } impl PDirectory { pub fn at(&self, idx: usize) -> PDirectoryEntry { self.entries[idx] } pub fn set_by_addr(&mut self, logical_addr: usize, entry: PDirectoryEntry) { self.set(PDirectory::to_idx(logical_addr), entry); } pub fn set(&mut self, idx: usize, entry: PDirectoryEntry) { self.entries[idx] = entry; unsafe { invalidate_page(idx); } } pub fn to_logical_addr(idx: usize) -> usize { (idx << 22) } pub fn to_idx(logical_addr: usize) -> usize { (logical_addr >> 22) } } use lazy_static::lazy_static; use spin::Mutex; lazy_static! { static ref PAGE_DIRECTORY: Mutex<&'static mut PDirectory> = Mutex::new( unsafe { &mut *(0xC0000000 as *mut PDirectory) } ); } pub unsafe fn setup_pd(pd: usize) { let mut data = PAGE_DIRECTORY.lock(); *data = &mut *(pd as *mut PDirectory); }

GRUB Memory Card and OS1 Physical Memory Card


In order to get a memory card from GRUB, I set the corresponding flag in the header during the boot phase, and GRUB gave me the physical address of the structure. I ported it from the official documentation to the Rust notation, and also added methods to conveniently iterate on the memory card. Most of the GRUB structure will not be filled out, and at this stage it is not very interesting to me. The main thing is that I do not want to determine the amount of available memory manually.


When initializing via Multiboot, we first convert the physical address to the virtual one. Theoretically, GRUB can locate the structure anywhere, so if the address is out of the page, you need to allocate a virtual page in the Page directory. In practice, the structure almost always lies next to the first megabyte, which we have already allocated at the boot stage. Just in case, check the flag that the memory card is present and proceed to its analysis.

A memory card is a linked list, for which the initial structure contains the initial physical address (do not forget to translate everything into virtual ones) and the size of the array in bytes. The list has to be iterated based on the size of each element, since in theory their sizes may differ. Here is the iteration: 


 impl MultibootInfo { . . . . . . pub unsafe fn get_mmap(&self, index: usize) -> Option<*const MemMapEntry> { use crate::arch::get_mb_pointer_base; let base: usize = get_mb_pointer_base(self.mmap_addr as usize); let mut iter: *const MemMapEntry = (base as u32 + self.mmap_addr) as *const MemMapEntry; for _i in 0..index { iter = ((iter as usize) + ((*iter).size as usize) + 4) as *const MemMapEntry; if ((iter as usize) - base) >= (self.mmap_addr + self.mmap_lenght) as usize { return None } else {} } Some(iter) } }

When parsing a memory card, we iterate through the GRUB structure and convert it into a bitmap with which OS1 will work to manage physical memory. I decided to limit myself to a small set of available values ‚Äã‚Äãfor management - free, busy, reserved, inaccessible, although GRUB and BIOS provide more options. So, we iterate over the map entries and convert their state from GRUB / BIOS values ‚Äã‚Äãto OS1 values: 


 pub fn parse_mmap(mbi: &MultibootInfo) { unsafe { let mut mmap_opt = mbi.get_mmap(0); let mut i: usize = 1; loop { let mmap = mmap_opt.unwrap(); crate::memory::physical::map((*mmap).addr as usize, (*mmap).len as usize, translate_multiboot_mem_to_os1(&(*mmap).mtype)); mmap_opt = mbi.get_mmap(i); match mmap_opt { None => break, _ => i += 1, } } } } pub fn translate_multiboot_mem_to_os1(mtype: &u32) -> usize { use crate::memory::physical::{RESERVED, UNUSABLE, USABLE}; match mtype { &MULTIBOOT_MEMORY_AVAILABLE => USABLE, &MULTIBOOT_MEMORY_RESERVED => UNUSABLE, &MULTIBOOT_MEMORY_ACPI_RECLAIMABLE => RESERVED, &MULTIBOOT_MEMORY_NVS => UNUSABLE, &MULTIBOOT_MEMORY_BADRAM => UNUSABLE, _ => UNUSABLE } }

The physical memory management takes place in the memory :: physical module, for which we call the map method above, passing it the address of the region, its length and state. All 4 GB of memory potentially available to the system and divided into four megabyte pages are represented by two bits in the bitmap, which allows you to store 4 states for 1024 pages. In total, this construction takes 256 bytes. The bitmap leads to terrible memory fragmentation, but this is understandable and easy to implement, which is central to my goal.


I will remove the implementation of the bitmap under the spoiler so as not to litter the article. The structure is able to count the number of classes and free memory, mark pages by index and address, as well as search for free pages (this will be needed later to implement the heap). The card itself is an array of 64 u32 elements; to isolate the necessary two bits (blocks), a conversion is used to the so-called chunk (index in the array, pack of 16 blocks) and a block (bit position in the chunk).


Physical memory bitmap 
 pub const USABLE: usize = 0; pub const USED: usize = 1; pub const RESERVED: usize = 2; pub const UNUSABLE: usize = 3; pub const DEAD: usize = 0xDEAD; struct PhysMemoryInfo { pub total: usize, used: usize, reserved: usize, chunks: [u32; 64], } impl PhysMemoryInfo { // returns (chunk, page) pub fn find_free(&self) -> (usize, usize) { for chunk in 0..64 { for page in 0.. 16 { if ((self.chunks[chunk] >> page * 2) & 3) ^ 3 == 3 { return (chunk, page) } else {} } } (DEAD, 0) } // marks page to given flag and returns its address pub fn mark(&mut self, chunk: usize, block: usize, flag: usize) -> usize { self.chunks[chunk] = self.chunks[chunk] ^ (3 << (block * 2)); let mask = (0xFFFFFFFC ^ flag).rotate_left(block as u32 * 2); self.chunks[chunk] = self.chunks[chunk] & (mask as u32); if flag == USED { self.used += 1; } else if flag == UNUSABLE || flag == RESERVED { self.reserved += 1; } else { if self.used > 0 { self.used -= 1; } } (chunk * 16 + block) << 22 } pub fn mark_by_addr(&mut self, addr: usize, flag: usize) { let block_num = addr >> 22; let chunk: usize = (block_num / 16) as usize; let block: usize = block_num - chunk * 16; self.mark(chunk, block, flag); } pub fn count_total(& mut self) { let mut count: usize = 0; for i in 0..64 { let mut chunk = self.chunks[i]; for _j in 0..16 { if chunk & 0b11 != 0b11 { count += 1; } chunk = chunk >> 2; } } self.total = count; } pub fn get_total(&self) -> usize { self.total } pub fn get_used(&self) -> usize { self.used } pub fn get_reserved(&self) -> usize { self.reserved } pub fn get_free(&self) -> usize { self.total - self.used - self.reserved } }

And now we got to parse a single element of the map. If a map element describes a section of memory that is less than one page in 4 MB or is equal to it, mark this page in its entirety. If more - we beat 4 MB into pieces and mark each piece separately through recursion. At the initialization stage of the bitmap, we consider all the memory areas unavailable so that when the card runs out, for example 128 MB, the remaining areas are marked as inaccessible. 


 use lazy_static::lazy_static; use spin::Mutex; lazy_static! { static ref RAM_INFO: Mutex<PhysMemoryInfo> = Mutex::new(PhysMemoryInfo { total: 0, used: 0, reserved: 0, chunks: [0xFFFFFFFF; 64] }); } pub fn map(addr: usize, len: usize, flag: usize) { // if len <= 4MiB then mark whole page with flag if len <= 4 * 1024 * 1024 { RAM_INFO.lock().mark_by_addr(addr, flag); } else { let pages: usize = len >> 22; for map_page in 0..(pages - 1) { map(addr + map_page << 22, 4 * 1024 * 1024, flag); } map(addr + (pages << 22), len - (pages << 22), flag); } }

Heap and management of her


Virtual memory management is currently limited only to heap management, since the kernel doesn‚Äôt really know anything else. In the future, of course, it will be necessary to manage the entire memory, and this small manager will be rewritten. However, at the moment all I need is static memory, which contains executable code and stack, and heap dynamic memory, where I will allocate the structures for multithreading. We allocate static memory at the boot stage (and generally limited to 4 MB so far, because the kernel fits in them) and in general there are no problems with it now. Also at this stage I do not have DMA devices, so everything is extremely simple, but understandable.


I gave a pile of 512 MB of the highest core memory space (0xE0000000), 4 MB below I store the heap usage map (0xDFC00000). To describe the state, I use a bitmap for physical memory, but there are only 2 states in it - busy / free. The size of the memory block is 64 bytes - this is a lot for small variables such as u32, u8, but perhaps optimally for storing data structures. Still, it is unlikely that we will need to store single variables on the heap, now its main purpose is to store the context structures for multitasking.


Blocks of 64 bytes are grouped into structures that describe the state of a whole 4 MB page, so we can allocate both small and large amounts of memory into several pages. I use the following terms: chunk - 64 bytes, pack - 2 KB (one u32 - 64 bytes * 32 bits per package), page - 4 MB. 


 #[repr(packed)] #[derive(Copy, Clone)] struct HeapPageInfo { //every bit represents 64 bytes chunk of memory. 0 is free, 1 is busy //u32 size is 4 bytes, so page information total size is 8KiB pub _4mb_by_64b: [u32; 2048], } #[repr(packed)] #[derive(Copy, Clone)] struct HeapInfo { //Here we can know state of any 64 bit chunk in any of 128 4-MiB pages //Page information total size is 8KiB, so information about 128 pages requires 1MiB reserved data pub _512mb_by_4mb: [HeapPageInfo; 128], }

When requesting memory from the allocator, I consider three cases, depending on the granularity:


  • A request for memory less than 2 KB came from the allocator. It is necessary to find a pack in which it will be free [size / 64, any non-zero remainder adds one] chunks in a row, mark these chunks as busy, return the address of the first chunk.
  • From the allocator came a request for memory less than 4 MB, but more than 2 KB. You need to find a page that has a free [size / 2048, any non-zero remainder adds one] packs in a row. Mark [size / 2048] of packs as occupied, if there is a remainder - mark [the remainder] of the chunks in the last pack as occupied.
  • A request for memory more than 4 MB came from the allocator. Find [size / 4 mi, any non-zero remainder adds one] pages in a row, mark [size / 4 mi] pages as busy, if there is a remainder, mark [remainder] of packs as occupied. In the last packet, mark the rest of the chunks as occupied.

Search for free sites also depends on the granularity - an array is selected for the search or bitmasks. Any OMA happens. In case of deallocation, a similar algorithm is applied, only for marking the released ones. Memory freed is not reset. All the code is big, I'll remove it under the spoiler.

Allocation and Page fault


In order to use a bunch, you need an allocator. Its addition will open for us a vector, trees, hash tables, boxes, etc., without which it is almost impossible to continue living. As soon as we plug in the alloc module and declare a global allocator, life will immediately become easier.


The implementation of the allocator is very simple - it simply refers to the mechanism described above. 


 use alloc::alloc::{GlobalAlloc, Layout}; pub struct Os1Allocator; unsafe impl Sync for Os1Allocator {} unsafe impl GlobalAlloc for Os1Allocator { unsafe fn alloc(&self, layout: Layout) -> *mut u8 { use super::logical::{KHEAP_CHUNK_SIZE, allocate_n_chunks}; let size = layout.size(); let mut chunk_count: usize = 1; if size > KHEAP_CHUNK_SIZE { chunk_count = size / KHEAP_CHUNK_SIZE; if KHEAP_CHUNK_SIZE * chunk_count != size { chunk_count += 1; } } allocate_n_chunks(chunk_count, layout.align()) } unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout) { use super::logical::{KHEAP_CHUNK_SIZE, free_chunks}; let size = layout.size(); let mut chunk_count: usize = 1; if size > KHEAP_CHUNK_SIZE { chunk_count = size / KHEAP_CHUNK_SIZE; if KHEAP_CHUNK_SIZE * chunk_count != size { chunk_count += 1; } } free_chunks(ptr as usize, chunk_count); } }
