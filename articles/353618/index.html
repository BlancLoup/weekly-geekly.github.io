User interaction


Evgeniy: In general, this is the most interesting part - how everything is rendered and where it breaks when the user does something.



Angular


We have an application, inside it there is a comments-widget, inside it there is a small comment-form. These are all components. What happens when a user clicks submit somewhere in the bottom component?

All Angular lovers adore zones. Zones are such a tricky thing that came out, oddly enough, from Dart. Zones implement monkey patch for all asynchronous functions of your browser: setTimeout, setInterval, as well as all user interactions, such as clicks.

When something happens, zones jerk and pull some code - this is a ChangeDetection. ChangeDetection runs from the root across all components, determines that something has changed, and renders everything. If, for example, there were 5 comments and the 6th one was added, then all 5 comments would be rebranded. So, but very quickly.

React


Ilya: Let's make the same React application. As we found out, it's not a problem to make applications with plus or minus the same architecture, but the matter is in the details.



We catch the submit event of the form, notify the component above, throw the action in the store, process the action. Store throw emit change. Emit change will arrive to all components that are associated with the store, and components will begin to check the props on the topic, the links to the store trees have changed or have not changed.



If they have changed, the component will call the rerender function, completely re-render the Virtual DOM, then Virtual DOM will compare its previous state with the current one, with the new one, and make a point change in the DOM tree.

Part 4. End


After a dispute, there is pain, and after pain - the end. Let's sum up some results of the battle of Angular against React.



Angular


  • Syntax. Angular is a beautiful and clear syntax. This is HTML, it works, and you do not need to learn strange JSX, only some stars, parentheses and so on. I see nothing wrong with that.
  • Languages. You can write on Angular in JavaScript, on TypeScript, on Dart, on Flow - on what you want.
  • Single maintainer. Google has Angular, Angular has Google - with all the pluses and minuses. Of the benefits - if Angular is updated, I can say for sure that all other packages that are included in Angular work, but, on the other hand, everyone knows that Google is Google.
  • Boxed solution. You can be sure that if you do a CLI install, then you will have everything you need - HTTP requests, tests, etc. And everything is out of the box and everything works. In principle, something can be replaced, but most people use the already prepared one with all the pluses and minuses.

React


  • Syntax. React is of course JSX. No, you can write without JSX, but I have not seen people who do that. You quickly get used to JSX and understand how convenient it is.
  • Languages. In the case of React, the most popular language is still JS. Now there is a small movement towards TypeScript, there are geeks who write on Dart, but this is not the mainstream at all.
  • Community . React has Facebook, which releases new versions of React, makes a completely new product React Fiber, which will have the same interface, but completely different insides. But React has a very large community, and you will most likely be able to build an application from ready-made components, connect wrapped familiar libraries and use it.
  • Constructor . React is just a template engine that recommends using the Flux approach, and then you either collect the whole ecosystem from the pieces yourself, or simply take a ready-made Starter-pack, which you like and draws many dependencies.
