The romantic merging of JSON files in Git

The merge operation (merge), performed by standard git tools, works well for text files that contain source code for programs. But the merging of text files containing rigidly structured data, in particular JSON, is a big headache.

To solve this problem, you can connect a separate merge tool for JSON files to git, which does not work line by line, but takes into account the structure of JSON objects.

I propose to use for this a javascript script that analyzes merged JSON files and makes a merge based on the structure and nesting of JSON objects.

What does this give?


For example, you keep your application settings in a JSON file. If at the same time two developers add a new parameter to the end of the file (each with its own name), then there will be a conflict when merging into git. The semantic merge script will figure out that two different parameters have been added and will merge automatically and without conflicts, including both parameters in the resulting file.

Also, semantic merging saves when JSON stores lists of objects with the same set of properties. If a new object is added to the middle of such a list, in which only a few property values ‚Äã‚Äãdiffer from those around you, the standard git merge, and any tool for line-by-line comparison is likely to be confused. You will have to resolve the conflict manually, carefully looking at the text and risking to spoil the JSON structure.

The same applies to changing the order of objects in JSON.

Semantic merger features


The order of named objects in files is ignored.
For unnamed objects (array elements), the order of the rows is important. Arrays are compared in the order of the elements of the array in both files. When adding an element to the middle of the array, this situation is not recognized and conflicts arise in all subsequent elements. Also, conflicts arise when adding elements to the end of an array in both merged files.

Conflicts


In case of a conflict (the same object is changed in both merged versions of the file), the information showing the essence of the conflict is added to the resulting merge file: 
{ "CONFLICT": "<<<<<<<<>>>>>>>>", //        "OURS": ... // ""   "THEIRS": ... // ""   "ANCESTOR": ... //     "PATH": ... //       ,   }


Script Versions


Since we are developing a windows-application and using git under windows, I had to finish the script so that it worked on windows out of the box, that is, through Windows Scripting Host. WSH does not support JSON, so the JSON parsing library is included directly in the script. For those who are ready to use node.js there is a more compact version of the script.

Instructions for enabling merge drivers in git


1. Put the jsonmerge.js script in the git \ lib folder, for example in % Program Files (x86)% \ Git \ lib \

2. We connect in git new merge-driver. To do this, make changes to the git configuration file.
To enable the driver for only one repository, make changes to the file <project folder> \. Git \ confg
Or, to enable the driver for all local repositories, make changes to the global configuration file .gitconfig , which lies in the user profile folder (in windows it is% userprofile%), for example C: \ Users \ <username> \. Gitconfig 

 [merge "json_merge"] name = A custom merge driver for json files driver = cscript //B //Nologo 'C:/Program Files (x86)/Git/lib/jsonmerge.js' %O %A %B recursive = binary

For node.js, the driver string looks like this: 
  driver = node 'C:/Program Files (x86)/Git/lib/jsonmerge.js' %O %A %B


Do not forget to change the path to the jsonmerge.js file to your own.

3. Specify for which file extensions to apply this driver in the .gitattributes file. It can be located in any project folder to extend its action to lower level folders. Usually located in the root folder of the project: 

 *.json merge=json_merge


Initially, the coffeescript script was taken from here , transferred to pure javacript and refined so that it could be run in windows via standard Windows Script Host. Error handling has also been added: if the JSON structure of one of the merged files is incorrect, the original script does not return a sign of conflict, and git considers the merge successful. This is fixed in my version.
