1.1. Applying breakpoints and modifying local variables


One of the most commonly used tools of the integrated debugger is a breakpoint (BreakPoint - hereafter BP). After installing BP, the program will work until it reaches a stopping point, after which its work will be interrupted and control will be transferred to the debugger.

The easiest way to install and remove BP is the F5 hot key (or its equivalent in the Debug-> Toggle breakpoint menu). There are other ways, but about them later.

After the program is stopped, we can examine the values â€‹â€‹of the local variables of the procedure in which the execution of the application was stopped, as well as analyze the stack of calls preceding the call of this procedure. Here we can change the values â€‹â€‹of these variables.

Where to put BP - of course there is no general answer. In fact, BP is designed to facilitate the study of the operation of the code, the correctness of which we are not sure of, or explicitly containing an error, which we cannot immediately detect.

It is much easier to set a stopping point and consistently execute each line of code than to spend hours studying the same code, trying to figure out where it started to work in a different way than we intended.

Let's consider the following example.

There is a task: to write code that will increase the value of the initially numbered variable by one 5 times and one more time by the number 123, after which it will display the result as a 10-bit and hexadecimal value. Expected values â€‹â€‹will be as follows: 128 and 00000080.

1.2. Trace (step by step debugging)


The essence of tracing is to step through each line of code.

Suppose we stopped at a preset BP, we analyzed the code and want to go to the next line. In principle, we can also put BP on it and run the program. And for the next, and for those after it.
Practically, exposing the BP on each line of the procedure code, we manually imitate what the debugger can do (in more detail in the second section).

And he can do the following:

  1. â€œTrace Intoâ€ command (â€œF7â€) - the debugger will execute the code of the current line of code and stop at the next one. If the current line of code calls any procedure, then the next line will be the first line of the called procedure.
  2. The â€œStep Overâ€ command (â€œF8â€) is similar to the first command, but the input to the body of the called procedure does not occur.
  3. The â€œTrace to Next Source Lineâ€ command (â€œShift + F7â€) is also a practically complete analogue of the first command, but is used in the â€œCPU-Viewâ€ window (this debug mode is not covered in the article).
  4. â€œRun to Cursorâ€ command (â€œF4â€) - the debugger will execute the program code up to the line where the cursor is located (with the condition that no other BPs were encountered during execution).
  5. â€œRun Until Returnâ€ command (â€œShift + F8â€) - the debugger will execute the code of the current procedure until it exits. (Often used as a counter-reception on a randomly pressed â€œF7â€ and also with the condition that no other BPs were encountered during the execution).
  6. In older versions of Delphi, the â€œSet Next Statementâ€ command is available, with which we can change the course of the program execution by setting any line of code as the current one. This feature is also available in the code editor where you can drag the arrow pointing to the current active line to a new position.

These teams do not require detailed consideration. Let's stop only on the â€œTrace Intoâ€ team (â€œF7â€).

For example, take the following code:

If we have a second module connected, then the â€œTrace Intoâ€ (â€œF7â€) command in this case will work as â€œStep Overâ€ (â€œF8â€).
