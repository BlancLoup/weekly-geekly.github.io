Features of Open Office Base 2.3

I myself am a great admirer of any Open Source projects, but the article will focus on the real disadvantages of the Open Office Base. I would just like to note that this is not inciting holivar, and I would love to hear advice on solving such problems.

The thing is, by the nature of my business, I often have to process quite large amounts of data. Quite large - ranging from 4 to 40 thousand records. We have to do it every month, because so far there is no other solution.

While we had the opportunity to use any software at work, there were no problems, because MS Access (as it turned out) is a multifunctional database with a wide range of capabilities, which unfortunately cannot be said about its OO Base counterpart.
And further in the text, the actual list of problems:

Not supported link tables
Every month I have to process a large dbf-file, which, after processing, must be given in the same form. So, as far as I understood, OO Base does not support the link of tables of a format other than the native one, as did Access (with one mouse click).

The solution, of course, was found: to do this, you have to create a separate database, connecting to the dBase source, and then put the dbf file in the directory specified when creating it.

Requests to modify data are not supported.
This is generally outrageous, and I personally do not understand. No, not that it was absolutely impossible to do it, the very possibility is. This is done via the Base-SQL menu:



But it cannot be saved for later execution.
Having read in various forums on OO, it turned out that such queries can work if you raise the SQL postgress on your machine

And subselekt also does not support. Perhaps this is the trouble of the dbf format tables, and maybe I didn‚Äôt check them all, but in dbf, it refuses to work on other conditions.

There is no possibility to insert many lines at once.
One of the most common operations in my work is a mass insert term in an existing table. In Base, this elementary function unfortunately not.
True, there is an alternative possibility consisting of the following steps:

1. Copy the required number of lines into the buffer, click on an empty space in Base, and execute the ‚ÄúPaste‚Äù command.

In addition to the obvious inconvenience, this option also works poorly with a large number of records.
Therefore, the insertion of records with the number starting from 3 thousand, hangs for about 20 minutes, after which Base unloads safely, without any changes in the table.
