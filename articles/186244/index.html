Mail.Ru rating: count more and better

Six months ago we told you what we did in the Rating in the winter . Summer is already nearing the middle, and, in addition to the visits in the reports about which we wrote here , we managed to do some more interesting things. Under the cut - complex reports, pictures in search queries, a new counter code and average daily indicators in statistics.

Sophisticated reports

In the spring, we made a new type of report in the Rating: it allows you to see data on visitors that meets the conditions specified by the user. Using this tool, the user can set one or more selection criteria and see the usual reports on visitors that meet this criterion (or criteria).



For example, you can see the statistics of site visits by men from Russia through mobile browsers. Within one criterion, you can specify a list of possible options. So, in browsers, you can choose Opera, Mozilla, mobile browsers; within this criterion will be selected visitors that match any of them.

The difference between the reports and the user-friendly sections is that reports on the selected criteria are compiled on request, and there is no need to create sections on the selected criteria in advance. In other words, you can set different sampling criteria and view the statistics on the sample after some time, even if several months have passed.

New counter code

As a result of lengthy discussions and decisive rebuff, the old Mail.Ru Rating counter code was retired at the end of May. He was a small JS-snippet, collecting information from the browser and immediately loading the logo or single-pixel gif. The new JS-snippet downloads the counter code (on new Firefox 3.6+ browsers, IE 10+, Chrome 2+, Safari 5+, iOS 5+, Android 3+ - asynchronously), and the already loaded code sends information to the server.

Over the years, various projects have asked how much our code slows down the page load. Now we can proudly answer - does not slow down at all (in new browsers, of course)!



Equally important is that this transition will allow you to update the counter code (for example, when adding new features). People are conservative: the counter code has changed many times over the 14-year history of the project, but many sites continue to use the code of many years ago (which we still support). Now, in order to update the counter code, webmasters do not have to update the snippet on their pages - it‚Äôs enough to do it once when switching to asynchronous code.

To summarize: the project has become a little bit better, began to count a little bit faster, and we are preparing new reports. Profit!
