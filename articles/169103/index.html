MPLS and VPLS on Mikrotik

On the one hand, the desire is somewhat strange - the organization of a ‚Äúserious‚Äù MPLS / VPLS on cheap Mikrotik-type hardware. On the other hand, for 70 bucks (1500-2000r) for the younger model RB / 750 (GL) we get a PE / CE device that can (among other things) L2VPN / L3VPN over an MPLS medium and can pump through about 70 megabits of duplex (on large packages).
Mikrotik RouterOS can do both MPLS (L3VPN, Traffic Engeneering) and L2VPN (l2circuit aka VPWS, VPLS), which covers almost all possible tasks (given the performance of iron, of course).

Interesting? I ask under the cat!

For a start, a little theory.

MPLS ( multiprotocol label switching ) - label- based routing mechanism. Each packet passing through the MPLS network, regardless of the type of this packet, is assigned a specific label on the basis of which the routing decision is made. It is important to note that the content is not studied, i.e. inside MPLS, you can drive absolutely any L2 / L3 traffic (STP, BPDU, etc., up to SDH and ATM) without any influence on the transport network (flood, Vlana with a large number of MAC addresses, loops and other "joys "Become you, as a transport, deep purple).

The key concept in MPLS is LSP ( Label Switched Path ), a packet path that is established based on the LDP, RSVP-TE, BGP or CR-LDP protocols. In this article I will not be distributed, according to what criteria is set LSP, and from the protocols I will focus on the LDP. The MPLS literature on the network is enough for an interested reader to learn a lot of new things, but the purpose of this article is to demonstrate the very specific features of RouterOS.

A router in an MPLS network can perform one of three functions:

1. CE-router (Client-edge) - is not a member of the MPLS network, and in general does not know anything about it, it is just a client device that has L2 / L3 connectivity with the provider‚Äôs router.
2. PE router (Provider-edge) - an edge MPLS router. It is the gateway for the client and forwards its traffic to the provider's MPLS network. Another name is LER - label edge router.
3. P router (provider router) - MPLS router that transparently passes MPLS traffic without analyzing its contents. Another name is LSR (label-switch router).

VPWS (l2circuit) is a point-to-point channel (aka pseudo-wire).
VPLS is a multipoint channel, for the client it looks like a virtual ‚Äúdumb‚Äù switch, transparently letting through any packets.

Now, when we have decided on terminology, we will consider concrete examples:

Miktorik configuration as a PE router

Raise interfaces and ip-addresses: 
 / interface bridge
 add name = loopback
 add l2mtu = 1526 name = vpn

 / ip address
 add address = 10.0.11.24 / 24 interface = eth2
 add address = 172.19.238.33 / 32 interface = loopback network = 172.19.238.33


We configure OSPF, announce networks through which we see our neighbors + loopback: 
 / routing ospf instance
 set [find default = yes] router-id = 172.19.238.33
 / routing ospf interface
 add interface = eth2
 / routing ospf network
 add area = backbone network = 10.0.11.0 / 24
 add area = backbone network = 172.19.238.33 / 32


Configuring LDP, don't forget about mpls-mtu (for successful passing of tagged packets and QinQ: 
 / mpls interface
 set [find default = yes] mpls-mtu = 1526
 / mpls ldp
 set enabled = yes lsr-id = 172.19.238.33 transport-address = 172.19.238.33
 / mpls ldp interface
 add interface = eth2


Finally, we raise L2VPN and wrap eth5 there, where our customers will live: 
 / interface vpls
 advertised-l2mtu = 1526 cisco-style = yes cisco-style-id = 5 disabled = no l2mtu = 1526 \
 name = junos-l2circuit remote-peer = 172.19.238.34

 / interface bridge port
 add bridge = vpn interface = eth5
 add bridge = vpn interface = junos-l2circuit

Cisco-style defines id parameters by which routers recognize each other. l2-mtu determines the maximum frame size that can pass without fragmentation.

Juniper SRX configuration as a P-router

Configuring interfaces, IP addresses: 
 interfaces {
     fe-0/0/6 {
         description mkt.192.168.168.2 - pe;
         mtu 1624;
         unit 0 {
             family inet {
                 mtu 1500;
                 address 192.168.168.1/24;
             }
             family mpls;
         }
     }
     fe-0/0/7 {
         description mkt.10.0.11.24 - pe;
         mtu 1624;
         unit 0 {
             family inet {
                 mtu 1500;
                 address 10.0.11.23/24;
             }
             family mpls;
         }
     }
     lo0 {
         unit 0 {
             family inet {
                 address 172.19.238.35/32;
             }
         }
     }
 }


Then we raise OSPF: 
 protocols {
     ospf {
         area 0.0.0.0 {
             interface fe-0/0 / 6.0 {
                 hello-interval 10;
             }
             interface fe-0/0 / 7.0 {
                 hello-interval 10;
             }
             interface lo0.0 {
                 passive;
             }
         }
     }
 }


Finally, configure MPLS: 
 protocols {
     mpls {
         interface all;
     }
     ldp {
         egress-policy CONNECTED-ONLY;
         transport-address 172.19.238.35;
         interface fe-0/0 / 6.0;
         interface fe-0/0 / 7.0;
         interface lo0.0;
     }
 policy-options {
     prefix-list LOOPBACK-PREFIX {
         172.19.238.35/32;
     }
     policy-statement CONNECTED-ONLY {
         from {
             prefix-list LOOPBACK-PREFIX;
         }
         then accept;
     }
 }
 security {
     forwarding-options {
         family {
             mpls {
                 mode packet-based;
             }
         }
     }
 }

In this case, SRX, as a P-router, does not install any MPLS connections itself, but only passes LDP packets.

Similar to the configuration for l2circuit, but there are a couple of nuances:
1. Channels in the same VPLS domain must have one vpls-id.
2. Connectivity should be full-mesh (each with each) to ensure the greatest reliability. However, this condition is not mandatory.
3. To get rid of the possibility of a loop, use the following option: 
  / interface bridge port set 0 horizon = 1
