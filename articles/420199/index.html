Automation of monitoring system based on Icinga2 and Puppet


Let's talk a little about ... Infrastructure as code (IaC).


On Habr√© there are some very good articles about Icinga2, there are also excellent articles about Puppet:

Icinga2 simple option

However, the topic of automation and integration of these two amazing systems is not disclosed at all.
In this tutorial, I will show you with a ‚Äúlive‚Äù example, by combining these two
system, get a powerful tool for monitoring your infrastructure with all the necessary functions. The article is a kind of guide to the installation of the package "all in one bottle." After completing this guide, you will have a fully operational monitoring solution that can be ‚Äúdoped‚Äù in the future. Let's try!

So:


We raised a new host. And we need:


1. To monitor it automatically appeared in Icinga2, and basic checks were created:


Checks
Snapshots
Explanations
Host



With a given frequency, we check with the ping command that the host is ‚Äúlive‚Äù.
Disk usage



Check that we have enough free disk space.
Load average



We monitor server load dynamically. It takes into account the number of processors on it.
Free memory



We check that we have enough free memory on the server.
Open ports



We scan ports on the server and create a map of open ports. We monitor that we did not have new open or closed ports.
Critical updates



We monitor the availability of critical updates on the server.


2. Add custom checks for various services in a convenient and understandable form. What would the director then show how good we are and get an award!


Some examples are:
Service
YAML verification code
Virtual host
Check the "live" host or not.

  '%{::fqdn} PostgreSQL': target: /etc/icinga2/zones.d/master/%{::fqdn}.conf apply: true assign: [ 'host.name == %{::fqdn}' ] display_name: 'PostgreSQL' command_endpoint: '%{::fqdn}' check_command: "postgres" vars: postgres_host: "localhost" postgres_action: "connection" phone_notifications: true

Nginx Status
We monitor the status of Nginx through stub_status. 




  '%{::fqdn} nginx status' : target: /etc/icinga2/zones.d/master/%{::fqdn}.conf apply: true assign: [ 'host.name == %{::fqdn}' ] command_endpoint: '%{::fqdn}' display_name: 'nginx status' check_command: 'nginx_status' vars: nginx_status_host_address: localhost nginx_status_servername: server.com nginx_status_critical: '1600,60,30' nginx_status_warn: '1500,55,25'


3. So that everything is neat, reliable and beautiful. And most importantly, spend no more than 30 minutes on the fuss with the initial setting.


You should have experience with Docker, and therefore with Linux - by itself.

This setup is described under Debian / Ubuntu. And, although I see no reason for him not to work on other Unix-like systems, I myself cannot give such guarantees. I have a couple of machines with CentOS, it works there, but most of them are Debian / Ubuntu.

Let's start


I will say right away, it is convenient for me when the entire configuration of the host is services, configs, software, accounts, etc. - are described by a single yaml file, this actually avoids documenting the infrastructure and gives a visualization of the configuration. Opened the corresponding project in the git-repository, where the file names correspond to the host name, then opened the configuration of the desired host. And you can immediately see what services are on the host, what is backed up from it, what is being monitored, etc.

The general scheme of such a scheme is just two simple steps:


  1. We start puppet on the host - the host sees the checks that belong to it and exports them to puppetDB.

  2. We start puppet in the icinga2 container - the checks from puppetdb are turned into real Icinga2 configs.
