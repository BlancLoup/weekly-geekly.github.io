Experience using SMI-S to automate StarWind iSCSI SAN Free management in System Center Virtual Machine Manager 2012 SP1

Due to the growing popularity of the Private Cloud idea, the task of automating the management of storage and hypervisors of virtual machines is of particular relevance. Microsoft offers to manage the park of Hyper-V machines using System Center Virtual Machine Manager (SC VMM) , but until recently, SC VMM did not have a fairly convenient way to automatically manage storage. By releasing SC VMM 2012 SP1 , Microsoft has eliminated this shortcoming by adding support for the SMI-S standard.

Now, from the VMM console, you can create and delete disks, assign and connect them to Hyper-V hosts, mark up and format them. Previously, this had to refer to the management utilities from the manufacturer of the disk array, which complicated the administration.

The purpose of this article is to demonstrate how to configure, connect, and use an SMI-S provider to automate the management of a disk array. Consider working with data storage using the SMI-S protocol using the example of the free iSCSI version of the StarWind iSCSI SAN repository. Archive with all necessary take here (there is already a new version, see. UPD below ).

SMI-S


Storage Management Initiative - Specification or simply SMI-S is a standard for managing disk storage. Its development was started by the association SNIA back in 2002. SMI-S is based on the Common Information Model (CIM) open standard and Web-Based Enterprise Management (WBEM) technology. At the moment, SMI-S is ratified as an ISO standard and is supported by many storage manufacturers.

The SMI-S specification is now regularly supported in Windows Server 2012 as part of Storage Spaces , which, for example, is mentioned here and here . In the context of automation, for us in Storage Spaces, the ability to use SMI-S in PowerShell scripts is important. About SMI-S support in SC VMM as applied to Microsoft iSCSI-Target has recently been a post on Habr√©.

The manufacturer of the disk array, of course, also needs to take care of SMI-S support in its products. He must provide a so-called. SMI-S provider, it is sometimes called the ‚Äúagent‚Äù, which plays the role of an intermediary in the interaction of the SMI-S client ( SC VMM ) and the server part of the disk array. The SMI-S client interacts with the SMI-S provider via the CIM-XML protocol, while the SMI-S provider itself can use proprietary mechanisms of interaction with the disk array.

Installing and configuring StarWind SMI-S Agent


StarWind company distributes SMI-S provider ( StarWind SMI-S Agent ) for free to manage a disk array based on their solution. At the moment, its beta version is available for download.

Archive with StarWind SMI-S Agent can be downloaded here (there is already a new version, see UPD below ). The archive also has the latest free version of StarWind iSCSI SAN , with which the SMI-S provider is compatible.

The installation itself is standard - the installer will ask a few questions, and then unpack the necessary files and register the system service. By the way, the StarWind SMI-S Agent itself does not have to be installed on the same machine where StarWind iSCSI SAN Free is installed.

After installing the StarWind SMI-S Agent, it must be configured. This is done using the StarWindSMISConfigurator utility (Figure 3). To start it, click Win + R , enter the StarWindSMISConfigurator command in the Run window, and then click OK .


Fig. 3

Think of the name of the disk array and enter it in the StorageSystemName field. In this case, he was given the name " TestStorage ". Specify the address of the host on which StarWind iSCSI SAN Free is running, do not change the control port number. Do not confuse the StarWind service management port (3261) with the port (5988), which will be accessible by the StarWind SMI-S Agent itself . In the path fields, you must enter the correct paths to the folders where the disk device image files will be stored. Paths in the path fields must be specified for the server hosting the iSCSI- tag. Specify them in the same format that is used when creating virtual devices in a StarWind iSCSI SAN .

ConcretePool # 2 does not need to be configured, since in the free version of StarWind there is no support for HA ( High Availability ) devices.

After you apply the new settings, restart the StarWind SMI-S Agent service .

Note. Always before restarting the StarWind SMI-S Agent , always verify that the StarWind iSCSI SAN service is up and running.
