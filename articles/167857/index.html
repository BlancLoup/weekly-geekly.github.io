CodeIgniter Application Development Standard

And we use internal standards. They are really useful:
- standards do not necessarily follow ;
- they allow faster and more comfortable to be mastered;
- they help to lose less in the creative process.

Historically, it happened so that we actively use CodeIgniter. The CodeIgniter application development standard is brought to your attention.


Controllers


1. One controller must be responsible for one function module.
2. Cross-functional modules can be allocated to separate controllers, or attached to existing ones. In both cases, the architectural solution should be the same for the entire product.
3. In each controller method, the functional zones should be taken into account and, if possible, separated:
- access rights;
- receiving and processing data;
- data output.
In this case, both the use of the _remap () method, and the extension of the controller class, and duplication of functional zones in each method are allowed. It is important that actions are explicitly defined and divided.
4. You should not use routing if this can be avoided.
5. You should not do data output from the controller, bypassing the class Output (display), if it is not AMF.
6. It makes sense to use private functions to shorten code with identical functionality.
7. The code of public methods of controllers should be high-level, concise and concise. Obviously, it makes sense to perform all the routine actions in private methods of controllers, models, mappings, helpers, or libraries.
8. It makes sense to use clear and concise method names, which will give beautiful and obvious URLs, which among other things contributes to more comfortable program maintenance.

Models


1. Each model performs actions only with its associated entity.
2. If possible, each entity should correspond to one table in the database. If, for servicing, for example, links, additional tables are required, it makes sense to consider the question: whether or not to make the link an additional entity with its model.
3. It makes sense to extend the class of models (example below).
4. It makes sense to use ORM.
5. It makes sense to avoid complex queries in favor of a series of simple queries that allow you to cache data and perform more flexible and faster processing.
6. Use Active Record.
7. You should not store in the database information that does not change when the program is executed.

Mappings


1. It makes sense to create and use an alias for $ this-> load-> view ($ view, $ data, TRUE ) in a procedural context (example below).
2. Mappings can load other mappings. You should use this feature:
- for organizing a hierarchical output structure
- to separate functional output logic and routines from templates
- to separate design templates. For example, it allows you to quickly and literally switch design
- for formatting structural and functional output (HTML page, JSON, HTML fragment, dynamic data)
3. It is not necessary to generate files and non-dynamic data in mappings.
4. In the mappings, you can change the format of the data. But it is not necessary to change the contents of the data in the mappings - for this there is the OOP-power of controllers, models and libraries, and also procedural helpers.
5. It is not necessary to use JavaScript in mappings, unless there is an obvious need for the opposite.

Configuration


1. It makes sense to create your own configuration files, since native ones tend to be updated along with the framework.
2. It makes sense to use subdirectories for functional configurations.
3. It is advisable to make in the configuration all the specific data and data that may change during the development of the project, rather than explicitly indicate them in the program code. This allows you to quickly find them, and comfortably modify.
4. For some configuration data that may change during the course of project development, it makes sense to provide a denormalization in the database to ensure the correctness of the history.

Libraries


1. You should not create a library if the need to create it is not obvious. For example, instead of a library, you can create a controller.
2. Libraries use object context, and this must be considered before its design.
General remarks
1. It is very useful to study, follow and reread at least once a year Style Guide.
2. It makes sense to use the solutions available in the framework before creating your own alternatives.
3. The framework is reliably super lightweight. This is confirmed by many tests and studies. It makes no sense to optimize it.
4. It makes sense to use debugging tools built into the framework (profiler and error logging modes).
5. It makes sense to place the application and system directories, as well as other files that should not be accessible by default to users of the site, above the Virtual Root Document Root.
6. Before designing something fundamentally new, it makes sense to peep in Google, on GitHub and BitBucket, the implementation of similar tasks. With a probability of 99%, any arbitrary task has already been solved. All trivial tasks have several solutions for CodeIgniter.
7. It makes sense to specify in autoload what is reliably often used in the current and in subsequent applications.
8. It is not necessary to specify specific resources in autoload that can be loaded in controller constructors.
9. Comments are good when they are relevant, concise and written in English.
10. TODO must be marked with the TODO keyword.
11. Before using third-party libraries, make sure they are written in PHP5 (and not PHP4)
