Upgrading to a new version of HP Application Lifecycle Management (ALM)

Hi Habr!

Every HP Application Lifecycle Management (ALM) administrator sooner or later will need to upgrade to a new version of HP ALM. Most of this process is the migration of HP ALM projects. Often, this is a rather trivial task, but sometimes there are moments that require further clarification.

In this article, I will review several scenarios for restoring HP ALM projects from a backup and a subsequent update.
So, your company decided to switch to a new version of HP ALM. After the euphoria has passed, it's time to start planning. For a start it is worth understanding: can you directly upgrade to the new version. HP ALM currently supports direct updates as follows:
v. 11.00 v. 11.5x v. 12.0x v. 12.2x
v. 10.x + +
v. 11.00 + + +
v. 11.5x + +
v. 12.0x +

If you are planning another transition, for example from v.10.xx to v.12.0x, this is also possible. To do this, you will need to do everything described below, but with an upgrade to an intermediate version of v.10.xx -> 11.5x -> 12.0x. Accordingly, the transition will just take a little more time.


The next step is to check if the version of the database server being used is supported by the new version of HP ALM. This information can be easily found on the HP website under HP QC / ALM . If the version of your database server is in the list of compatible ones, and you do not plan to switch to the new version (we will look at the opposite case a little later), proceed directly to the update.

For a successful update, I would recommend a smooth transition, with a consistent update of projects. In case you have a lot of active projects, you need to be safe in case of unforeseen situations. The abrupt transition, namely the update of all projects as part of the product installation, can stop the work of the whole company.

The first step is to perform a bureaucratic procedure to obtain resources for the new server machine.
Next, we proceed with the installation of HP ALM, and, as always, we have several options:

  • Updating the database of the entire product and projects
  • Clean installation, and subsequent migration and updating of projects on a new product base

In both cases there are some nuances, for example, when updating the product database, all your projects and settings will appear on the new server, so backup copies must be made in advance for all projects. In the second case, your settings and, most importantly, passwords will not appear on the new server. I‚Äôm a fan of LDAP authentication, so it doesn‚Äôt scare me. You can also talk to the database administrator, I think - he will figure out how to transfer passwords to a new server.

If you are planning a smooth transition, I would recommend a clean installation, but, as always, with the caveat: to make your life easier, use the same paths to file storages, passwords and connection strings (English connection string) were used in the previous version, but be careful not to catch the "productive". As for the project file storage, I recommend switching to network storage if you are still not using it.

The next step is to select a project for migration to the new version. Migration should begin by disabling the selected project in the administration website. If you are using version control, do not forget that all entities must be Check-ed In (returned). Next, create a copy of the project database using the database server administrator and a copy of the project file storage.



You are almost ready, but if you are using MS SQL, you still need to run the following script for a copy of the project database: DIRECTLY : 

EXEC sp_change_users_login 'Report' EXEC sp_change_users_login 'Update_One', 'td', 'td' EXEC sp_changedbowner   'sa'

This will ensure that the database user used by the product can work with the restored database.
