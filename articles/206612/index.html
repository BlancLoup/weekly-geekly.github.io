SNMP MIBs and how to cook them

Good day, reader.

Prehistory


I have been working for 5 years in a company that provides broadband access to the world wide web. I am engaged in direct control of equipment, as well as network planning and monitoring. At the time of my arrival in the network there were a little more than 100 pieces of equipment managed by vendors Cisco and D-Link. We drew a network map in the old-fashioned way in visio, ip-addresses were in the same place, but at that time my manager began to write some software for inventory of equipment - it all started with taking into account the issued ip-addresses. Given the growth of the network (or rather, modernization in the face of replacing stupid soap boxes with ‚Äúcontrollers‚Äù), we decided that we lacked the usual Cacti. We collected standard SNMP counters right away, but it was difficult to find something more necessary and interesting. Repeated attempts to master other options did not lead to the final result, but not so long ago we managed to ‚Äúcurb‚Äù the MIB files for D-Link switches and I believe that this article will speed up the process of studying switches and the SNMP protocol. This will allow faster and better network management.


Installing MIBs


Standard

MIBs are usually distributed as an archive with a bunch of files. Many of them, compiled in iana and ietf, are repeated in each archive, but transmitted for compatibility.
To work in the default system (specifically for Debian), they should be approximately in / usr / share / mibs
First, install the standard mibs in the system. 

$ sudo apt-get install snmp-mibs-downloader

In the configuration file /etc/snmp/snmp.conf include the necessary. Example: 
 % cat /etc/snmp/snmp.conf # # As the snmp packages come without MIB files due to license reasons, loading # of MIBs is disabled by default. If you added the MIBs you can reenable # loaging them by commenting out the following line. mibs :BRIDGE-MIB:Q-BRIDGE-MIB:IP-MIB:IF-MIB #mibs :ALL


mibs: ALL includes everything that is not quite good. I recommend for each equipment to have a folder with mibs, since They may differ from one firmware to another.

Software


D-view


The first attempt to use these MIBs was made in the D-View utility from the switch manufacturer. The package includes MIB Browser and MIB Compiler. After some attempts, I managed to see the tree for the DGS3120-24SC-L2MGMT-MIB module, but there was no search by oid and when loading the module, you need to manually specify several additional MIB files, which slowed down the work. Everything becomes complicated if you have only a numeric oid on your hands and you don‚Äôt know which of the MIB modules it can be - you will have to download all the MIBs from the folder one by one and check each one. At this point I finished studying the D-View program - it does not suit me.
Net-snmp

Returning to where I started the post:
We downloaded the archive from the MIBs and will use the snmptranslate utility from the Net-SNMP package. For convenience, we put all the mibs in one directory, but this is still not enough: 
 % snmptranslate -M ./ -m DGS3120-24SC-L2MGMT-MIB -Ln 1.3.6.1.4.1.171.11.117.1.3.2.100.1.2.0.1 iso.3.6.1.4.1.171.11.117.1.3.2.100.1.2.0.1

In order not to suffer for a long time, copy the missing files from the des-3200 switch mibs with the option not to overwrite the existing ones. And here we already get a positive result: 
 % snmptranslate -M ./ -m DGS3120-24SC-L2MGMT-MIB -Ln 1.3.6.1.4.1.171.11.117.1.3.2.100.1.2.0.1 DGS3120-24SC-L2MGMT-MIB::swL2macNotification

UPD: You can not copy files, and specify the directory local and standard from the system -M ./:/usr/share/mibs/ietf:/usr/share/mibs/iana (for convenience, you can do alias in the shell)

About flags:
  • -M Indicates the directory where to find mibs (can be listed, separated by ":")
    Thus, it was possible not to add all MIBS to one directory, but then you need to specify all directories from the original archive with the -M flag.
  • -m Specifies which module to activate (can be listed, separated by ":")
    Without the indication of the module, those specified in /etc/snmp/snmp.conf will be used
    % snmptranslate -m ./ -Ln 1.3.6.1.4.1.171.11.117.1.3.2.100.1.2.0.1
    RFC1155-SMI :: enterprises.171.11.117.1.3.2.100.1.2.0.1
  • -Ln Delivers us from the infinite footwoman of errors in the mibs hierarchy. More precisely -Ln disables errors in stdout


Now that the translation works, you can taste the beauty of the OIDs hierarchy. For this there are flags: 
 -Td Print full details of the specified OID. -Tp Print a graphical tree, rooted at the specified OID. -Ta Dump the loaded MIB in a trivial form. -Tl Dump a labeled form of all objects. -To Dump a numeric form of all objects. -Ts Dump a symbolic form of all objects. -Tt Dump a tree form of the loaded MIBs (mostly useful for debugging). -Tz Dump a numeric and labeled form of all objects (compatible with MIB2SCHEMA format)

D-Link Pitfalls

There is also a feature in mibs for 3200, that Object Name loop_detect is present in them. The "_" symbol in net-snmp is not supported, so for prevention we do:% sed -i 's / _ / - /' * in the directory with mibs. You can see an error about this symbol if you return error output (remove the -Ln key): 
 % snmptranslate -M ./ -m DES3200-10-L2MGMT-MIB -Td 1.3.6.1.4.1.171.11.113.2.1.2.100.1.2.0.1 No log handling enabled - turning on stderr logging Expected "(" (_): At line 5892 in .//L2mgmtDES3200-10.mib Should be ACCESS (detected): At line 5892 in .//L2mgmtDES3200-10.mib Bad parse of OBJECT-TYPE: At line 5892 in .//L2mgmtDES3200-10.mib SWPRIMGMT-DES3200-MIB::des3200-10-cx.2.100.1.2.0.1 des3200-10-cx OBJECT-TYPE -- FROM SWPRIMGMT-DES3200-MIB ::= { iso(1) org(3) dod(6) internet(1) private(4) enterprises(1) dlink(171) dlink-mgmt(11) des3200SeriesProd(113) des3200-10(2) des3200-10-cx(1) 2 100 1 2 0 1 }

I think the vast majority of libraries use the MIBs and the /etc/snmp/snmp.conf config for OID (the Net-SNMP uses them, and the library refers to the latter), but we don‚Äôt want to include these MIBs globally.
