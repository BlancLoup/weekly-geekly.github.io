Redis is an advanced key-value repository. Experience of use in loaded projects

We have been successfully using Redis in our highload projects for half a year. During this time, many successful Redis solutions have been developed for the tasks faced by the developers of most large projects. I would like to share examples of these decisions on the upcoming DEVCONF , in order to show in practice the possibilities, advantages and weaknesses of Redis.

Redis - a brief overview (or Redis is not just a key-value database):
  • Redis is very fast
  • Data types: String, List, Set, Sorted set, Hash
  • Atomicity of operations
  • Data Preservation: Snapshot, Append-Only File (binlog)
  • Transactions
  • Virtual memory
  • Master / Slave replication
  • Publish / Subscribe (message queues)
  • Pipeline
  • Redis tools
  • Clients (PHP, Ruby, Python, Erlang, Perl, Tcl, Java, Lua, C #, Javascript / Node.js, ...)

Comparison of performance and features with other repositories:
  • MySQL (InnoDB)
  • Memcached
  • Cassandra
  • Tokyo Cabinet
  • MongoDb

Learn more about features with practical examples:
  • Key space
    Separate key storage on a single Redis server. Base for testing and development environment.
  • String-value keys
    Unique identifiers, counters, serialized objects.
  • Key lifetime
    Caching, session storage.
  • Lists - ordered list of values
    Keeping a list of keys of serialized objects, simple queues.
  • Sets - an unordered set of unique values
    Tags, friends, pending deletes, online users list, ‚Äúvisited page‚Äù.
  • Sorted Set - a set of unique values ‚Äã‚Äãordered by weight
    Voices and user rating, autocomplex.
  • Hashes
    A new more suitable way to store and work with serialized objects.
  • Transactions
    Integrity and atomicity of the execution of a set of commands.
  • Publish / Subscribe
    Queue, chat.
  • Pipeline
    Quick execution of a set of commands.
  • Fault tolerance and data integrity
    Uninterrupted operation and guaranteed data integrity. Master / Slave replication.
  • Key Blur on Redis Servers
    Rediska: crc32, consistent hashing, cloud.
  • Service and monitoring
    Redis Tools. Commands INFO and MONITOR.

In conclusion, I will talk about upcoming innovations, as well as where, how and why you should not use Redis.
